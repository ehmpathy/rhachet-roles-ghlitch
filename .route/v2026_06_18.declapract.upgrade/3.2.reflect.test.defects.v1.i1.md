# 3.2.reflect.test.defects

## classification

---

### defect 1: Error cause type incompatibility

**classification**: practice bug

**rationale**: the declapract practice adds `{ cause: error }` pattern to error throws via jest.integration.env.ts but does not ensure tsconfig.json includes ES2022 in the lib array. the Error.cause feature requires ES2022.

**practice location**: jest.integration.env.ts template

**recommendation**: update practice to either:
1. add ES2022 to tsconfig.json lib array
2. or document that repos must have ES2022 support

---

### defect 2: env var name mismatch

**classification**: practice bug (now fixed upstream)

**rationale**: the prior local jest.unit.env.ts used old env var names (`STAGE`, `I_KNOW_WHAT_IM_DOING`) that predate the declapract standardization on `CONFIG` and `I_KNOW_THE_RISKS`.

**practice location**: jest env file templates

**recommendation**: upgrade to latest declapract version to receive the standardized env var names. the practice has been updated.

---

### defect 3: dpdm cycles in node_modules

**classification**: practice bug

**rationale**: the new `test:lint:cycles` command runs dpdm with `--transform` which follows imports into node_modules. this detects internal cycles in third-party packages (bottleneck, aws-sdk) that we cannot fix. the practice does not document how to handle these.

**practice location**: package.json test:lint:cycles command, .dpdmrc.yaml template

**recommendation**:
1. document that .dpdmrc.yaml may need to exclude third-party packages with known internal cycles
2. provide examples of common packages with internal cycles (bottleneck, aws-sdk)
3. clarify that exclusions are acceptable for cycles INTERNAL to third-party packages, but never for cycles in our own code or cycles we create with dependencies
