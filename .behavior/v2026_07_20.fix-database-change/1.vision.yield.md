# vision — expose the change class via provision.database (sync + briefs) 🗄️⛵

> sql-schema-control tracks two classes: **changes** (adhoc sql, recorded in a changelog table by id+hash) and **resources** (declarative DDL, diffed against live state). `provision.database` today speaks only the resource dialect — whole-config `plan`/`apply`.
> the change dialect has one operation left unexposed — **`sync`** (reconcile the changelog for a change applied out-of-band, without a re-run) — and one pattern left untaught: **a `MANUAL_MIGRATION` ALTER belongs in a `type: change` definition**, so normal `apply` runs it once and records it in the changelog.
> decision (wisher): **all changes flow through plan/apply control.** expose `--mode sync`, add briefs that teach the change pattern, and retire the per-repo raw-file bypass. no direct-file-apply escape hatch.

---

## the problem, grounded

sql-schema-control's `apply` converges **resources** by diff. but when a resource needs an ALTER it cannot auto-generate, it emits `MANUAL_MIGRATION` and **skips** it (`getRequiredActionForResource.ts:28`, with a literal `TODO - support automated migrations`). the readme shows the skip: `↓ [MANUAL_MIGRATION] … [skipped]` (`readme:153`).

two paths exist today, both outside `provision.database`:

| path | example | cost |
|---|---|---|
| **raw file apply** (retire) | svc-notifications' `provision.schema.migrate.ts` — raw `pg` client, `BEGIN`/run/`COMMIT`, **no changelog write** | untracked, re-runs unguarded, drifts from control, per-repo, "migrate" termsmell |
| **reconcile the changelog** | drop to `npm run provision:schema:sync -- --slug` | outside the family, ungated, undiscoverable |

the idiomatic answer sql-schema-control already offers: a DDL ALTER can be a **`type: change`** definition (`readme:41` notes change definitions cover the full range from user creation to table alteration). then `apply` runs it once and records it — "adhoc application of database changes tracked against the changelog," exactly the wish. so the ALTER never needs a raw bypass; it needs to be a change.

## the outcome world

### day-in-the-life — resolve a MANUAL_MIGRATION (through control)

`plan` flags a resource ALTER it won't auto-apply:

```
$ rhx provision.database --which livedb --env prep --mode plan
   ↓ [MANUAL_MIGRATION] ./tables/voice_call.sql (resource:table:voice_call) [skipped]
```

instead of a raw one-off apply, you express the ALTER as a **`type: change`** definition in `control.yml`, then run the normal flow — sql-schema-control runs it once and records it in the changelog:

```
$ rhx provision.database --which livedb --env prep --mode plan     # previews the new change
$ rhx provision.database --which livedb --env prep --mode apply    # runs + records it
```

a re-run is a no-op — the changelog remembers. no drift, no double-apply. (a brief walks through this.)

### day-in-the-life — reconcile a hand-applied change (`sync`)

a change was run by hand in an incident; the changelog is now stale, so the next `apply` would re-run it. reconcile it without a re-run:

```
$ rhx provision.database --which livedb --env prep --mode sync --slug 2026_07_20.voice_call_duration
```

```
🐈 smooth sailin!

⛵ provision.database --which livedb --env prep --mode sync --slug 2026_07_20.voice_call_duration
   ├─ change: 2026_07_20.voice_call_duration
   └─ changelog reconciled (no sql executed)
```

### before / after

| | before | after |
|---|---|---|
| a MANUAL_MIGRATION ALTER | per-repo raw file apply (untracked, drifts) | a `type: change` def → `apply` runs + records it (a brief teaches it) |
| reconcile a hand-applied change | raw `npm run provision:schema:sync -- --slug` | `provision.database --mode sync --slug <slug>` |
| the term | "migrate" (overloaded termsmell) | **change** — sql-schema-control's own ubiqlang |
| the raw bypass | svc-notifications' bespoke skill | retired — all changes flow through control |

### the "aha"

there is no special "apply this file" door. **the change class already is the door** — express the adhoc DDL as a change and `apply` records it. the only absent skill op is `sync`, for the one case that lives purely in the changelog. one mental model (change vs resource) covers it all.

## user experience

### usecases & goals

| who | goal | how |
|---|---|---|
| deployer | preview / apply the declarative config | `--mode plan` / `--mode apply` (unchanged) |
| deployer | apply an adhoc DDL change, recorded in the changelog | express it as a `type: change`, then `--mode apply` (a brief teaches it) |
| deployer | reconcile a change applied out-of-band | `--mode sync --slug <change-slug>` (**new**) |
| newcomer | learn change-vs-resource, and resolve a MANUAL_MIGRATION | read the new deployer briefs |

### contract — inputs & outputs

one skill, artifact ⛵ (deployer). `--mode` gains `sync`; no other part of the contract changes:

```
rhx provision.database --which livedb --env <env> --mode plan
rhx provision.database --which livedb --env <env> --mode apply
rhx provision.database --which livedb --env <env> --mode sync  --slug <change-slug>   # new
```

| arg | modes | denotes |
|---|---|---|
| `--which` | all | database target: `livedb` (unchanged) |
| `--env` | all | `prep` \| `prod` (required, no default) |
| `--mode` | all | `plan` \| `apply` \| **`sync`** |
| `--slug` | **sync only** | the change definition's natural key — the `slug:` field in `control.yml` (required for sync; forbidden on plan/apply) |

guarantee (unchanged shape): exit 0 = done; exit 1 = malfunction; exit 2 = constraint (absent/invalid args — e.g. `sync` without `--slug`, `--slug` on plan/apply).

under the hood `sync` maps to the consumer's `provision:schema:sync` run-command (`npx sql-schema-control sync -c <control.yml> --slug <slug>`), as `plan`/`apply` map to their run-commands. **on the flag name:** our contract exposes `--slug` (the ubiqlang-correct term for a human-authored natural key). sql-schema-control accepts `--slug` directly — the skill passes it straight through, no translation. (a handoff at `refs/handoff.sql-schema-control.rename-id-to-slug.md` tracks the upstream rename of the change field from `id` to `slug`.)

### timeline — resolve a MANUAL_MIGRATION

```
plan flags MANUAL_MIGRATION  →  write the ALTER as a type: change def  →  plan (preview)  →  apply (runs + records)  →  changelog remembers  →  future applies skip it
```

## mental model

**how a user describes it to a friend:**
> "sql-schema-control auto-manages *resources* (tables, functions) but punts on ALTERs it can't figure out — it flags them and skips. the fix isn't a special apply command; you just write the ALTER as a *change*, and the normal apply runs it once and logs it. `provision.database` now also lets you *sync* the log when someone already ran a change by hand."

**analogies:**
- **resources = declarative (terraform); changes = imperative one-offs (a tracked migration)** — one config holds both.
- **`sync` = liquibase `changelogSync`** — the tool cites liquibase as inspiration.

**terms — theirs vs ours (sql-schema-control ubiqlang):**

| they say | we say |
|---|---|
| "migration" / "the migrate skill" | an **adhoc change** (`type: change`); avoid "migrate" (overloaded termsmell) |
| "the ALTER it skipped" | a **MANUAL_MIGRATION** (its status literal) |
| "the ledger of applied changes" | the **changelog table** |
| "mark it already done" | `--mode sync --slug <slug>` |
| "the change's slug in control.yml" | our `--slug` (ubiqlang: a natural key, not a db-generated id); the tool accepts `--slug` directly |

## evaluation

### how well it solves the goals

| wish goal | met? | how |
|---|---|---|
| expose adhoc changelog-tracked change behavior | ✅ | `sync` exposed; the apply-a-change path is already `apply` on a `type: change` def (taught by a brief) |
| tracked against changelog, not declarative resource state | ✅ | changes record the changelog; the retired raw bypass did not |
| cover svc-notifications' migrate usecase | ✅ | resolved properly — a MANUAL_MIGRATION becomes a `type: change` (brief); no raw mode needed |
| "a brief or two" | ✅ | **two, cross-linked**: (1) change-vs-resource + the changelog; (2) handle a MANUAL_MIGRATION with a `type: change`, and when to `sync` |
| avoid the "migrate" termsmell | ✅ | ubiqlang = **change** |

### pros

- **one mental model** (change vs resource); no bespoke escape hatch to maintain.
- **tracked + idempotent** by construction — changes record the changelog; no drift.
- **prod-gated** `sync` (a changelog write is a prod mutation), consistent with the deployer-locks model.
- **retires** a per-repo untracked workaround; the family absorbs the usecase the right way.

### cons / tradeoffs

- **teach, don't tool.** the migrate usecase is covered by a brief + the change pattern, not a new command. someone who wanted a one-keystroke "apply this file" gets a pattern instead. accepted — the wisher chose control-only.
- **svc-notifications must move off** its raw skill (coordination in another repo; out of this repo's direct control).
- **the sql-schema-control gap remains.** it still cannot auto-apply resource ALTERs (`getRequiredActionForResource.ts:28` TODO). the `type: change` pattern is the workaround-of-record; if a resource ALTER truly cannot be a change, that is an upstream limitation, out of scope here.

### edgecases & pit-of-success

| edgecase | behavior |
|---|---|
| `--mode sync` without `--slug` | blocked, exit 2 |
| `--slug` on plan/apply | blocked, exit 2 |
| `--mode sync --env prod` (no grant) | gated by `provision.uses` — a changelog write is a prod mutation |
| `--mode plan --env prod` | ungated — preview only reads |
| consumer repo lacks `provision:schema:sync` | malfunction, exit 1, with a hint |
| unknown change id | sql-schema-control error surfaces (exit 1, failfast) |

`--slug` required-for-sync / forbidden-elsewhere makes the illegal states unrepresentable.

## what is awkward?

1. **the fix is a lesson, not a lever.** the most visible pain (a MANUAL_MIGRATION ALTER) gets no dedicated command — the answer is "express it as a change." correct per the control-only decision, but it asks the user to learn a pattern rather than press a button. the brief must carry real weight.
2. **cross-repo retirement.** svc-notifications owns the raw skill; we can teach the replacement and expose `sync`, but we cannot delete their skill from here. their move off it is a follow-up, not in this wish's blast radius.
3. **an upstream gap peeks through.** sql-schema-control itself flags `TODO - support automated migrations` for resource ALTERs. the `type: change` pattern is the workaround-of-record; a resource ALTER that resists a change model would expose that gap — but that is sql-schema-control's problem, not this skill's.
4. **`--which livedb` stays vestigial.** it only ever accepts `livedb`; `sync` inherits it for contract uniformity. a prior quirk, not new debt.
