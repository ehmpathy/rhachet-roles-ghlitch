# 1.vision — fix use.vpc.tunnel: env-correct tunnels for test/prep/prod

## the wish, distilled

> ensure our tunnel supports both prep and prod and test, correctly.

the handoff worried ghlitch had the same bug as svc-notifications: a hardcoded
`TUNNEL_CONFIG` map with every env aimed at `.prod`.

### the fix

ghlitch's shape differs from svc-notifications (it has **no hardcoded env→host map** —
so the exact "every env aimed at prod" bug can't occur here), but the wish still lands
squarely: **`use.vpc.tunnel` is not env-aware and must be.**

today the standalone `use.vpc.tunnel.sh` takes raw `--bastion`/`--cluster`, defaults
port to `5432`, and unlocks `--env all` — the operator hand-selects env-specific values
with no guard against a mismatch. meanwhile the peer `use.rds.capacity` is **already
env-correct**: it takes `--env`, reads per-env targets from `getConfig`, and fails fast.
the tunnel skill is simply the odd one out.

so the wish reduces to:

> make `use.vpc.tunnel` env-aware — take `--env test|prep|prod` and derive its target
> per-environment the same correct way `use.rds.capacity` already does — so a prep
> session can never be aimed at prod, and all three envs work correctly.

## the outcome world

### day-in-the-life

an operator (or an integration test, or CI) wants a database tunnel to prep:

```bash
rhx use.vpc.tunnel --env prep
```

the skill unlocks keyrack for **prep only**, reads the **prep** tunnel target from
the consuming repo's `getConfig()`, opens the ssm tunnel to the prep bastion +
prep cluster, and binds the prep host alias. there is no way to pass a bastion that
belongs to a different env than the creds, because the operator never types a
bastion — the env is the single source of truth.

### before / after

| | before (today) | after (wish fulfilled) |
|---|---|---|
| entry | `rhx use.vpc.tunnel --bastion X --cluster Y --port 5433 --host Z` | `rhx use.vpc.tunnel --env prep` |
| env selection | none — operator hand-types env-specific values | `--env test\|prep\|prod`, required, no default |
| target source | raw cli args (easy to mismatch env) | per-env `getConfig()` in the consuming repo |
| creds | `keyrack unlock --env all` (unlocks everything) | `keyrack unlock --env $ENV` (scoped) |
| port | defaults to `5432` | comes from config; no default invented |
| absent config | proceeds, fails opaquely downstream | fails fast: `belay that` + named keys + hint |
| parity with use.rds.capacity | divergent | identical config contract |

### the "aha" moment

the value clicks when an operator runs `rhx use.vpc.tunnel --env prep` and a prod
tunnel is now **hard to open by accident**. the env name is the only knob in the
common path; each per-env target value (bastion, cluster, host, port, account) is
derived from it via config. (region is the lone exception — currently hardcoded
`us-east-1`, not env-derived; see assumptions.) the same `--env` knob already governs
`use.rds.capacity`, `use.testdb`, and every other operator skill — so the tunnel is no
longer the odd one out.

> ⚠️ honesty check (surfaced in review): this "hard to open by accident" claim only
> holds if `--env` is the **sole** target selector. as long as raw `--bastion` /
> `--config` remain, the mismatch door stays ajar. the two are in tension — see the
> design stance below.

## user experience

### usecases / goals

- **open a tunnel to prep** for manual db exploration → `--env prep`
- **open a tunnel to prod** for an incident → `--env prod`
- **wake + tunnel to a serverless cluster** before tests → `use.rds.capacity --env test|prep|prod` (already correct; shares the engine)
- **never** accidentally point a prep session at the prod cluster

### contract inputs & outputs

inputs:
- `--env test | prep | prod` — required, validated, no default (mirrors `use.rds.capacity`)
- `help | --help | -h`
- (removed by design stance: raw `--bastion`/`--cluster`/`--port`/`--host`; `--config`
  retained only if the wisher names a real non-repo caller — see open-q #3)

outputs (treestruct, ghlitch cat vibes):
- success → `🐈 caught it!` + tunnel host/port
- absent/invalid env → `🐈 belay that...` + hint, exit 2
- absent config key → `🐈 belay that...` + named keys (`database.tunnel.bastion.exid`, `database.tunnel.cluster.name`, `aws.account`) + hint, exit 2
- aws/ssm failure → exit 1

exit semantics (already declared in the skill header): `0` active, `1` malfunction, `2` constraint.

### what it looks like to leverage

```bash
# prep tunnel
rhx use.vpc.tunnel --env prep
# 🐈 caught it!
# 🦺 use.vpc.tunnel --env prep
#    └─ tunnel open: aws.ssmproxy.ahbodedb.prep:15432

# forgot the env
rhx use.vpc.tunnel
# 🐈 belay that...
#    ├─ absent required arg: --env
#    └─ hint: rhx use.vpc.tunnel help   (exit 2)

# repo config never filled in for prod
rhx use.vpc.tunnel --env prod
# 🐈 belay that...
#    ├─ absent: database.tunnel.bastion.exid
#    └─ hint: set these in your repo config for prod   (exit 2)
```

### timeline

one uniform path, config-driven:
unlock keyrack (scoped to env) → export scoped aws creds → read per-env config →
fail fast if config absent → **if the config's tunnel host is `localhost` (no real
bastion): skip the ssm tunnel, point at localhost** → else `declastruct apply` opens the
ssm tunnel → confirm.

test falls out of this naturally: `getConfig()` models test as
`local: { host: 'localhost', port: 7821 }` (`use.testdb.sh` provisions there), so the
localhost branch triggers **from the config**, not from a hardcoded `env == test` check.
env values stay in config — consistent with the whole thesis.

### design stance (confirmed with wisher)

1. **`--env` is the only target selector.** drop raw `--bastion`/`--cluster`/`--port`/
   `--host`. the pit-of-success and raw-arg flexibility are mutually exclusive; env-as-
   source wins. keep `--config` only if a real non-repo caller surfaces; default remove.

2. **localhost short-circuit is config-driven, not env-hardcoded.** the skill reads the
   env's tunnel config; **if the host is `localhost` (no real bastion), it skips the ssm
   tunnel and points at localhost.** test satisfies this because `getConfig()` models
   test as `local.host = 'localhost'`. no `if env == test` special-case — the config
   decides, so the skill stays uniform and env values stay in config.

3. **`use.rds.capacity` composes `use.vpc.tunnel`.** today `use.rds.capacity.sh`
   *duplicates* the tunnel work (reads getConfig, exports `VPC_TUNNEL_*`, drives the
   engine directly — lines 109-150). once `use.vpc.tunnel --env` exists, `use.rds.capacity`
   should just call `rhx use.vpc.tunnel --env $ENV` then do its own `pg_isready` wait.
   this **deletes the duplicated config-read** and makes the relationship a clean
   composition: `use.rds.capacity` = open tunnel (delegate) + await capacity (its job).
   this also answers the DRY question — the shared logic lives in `use.vpc.tunnel`, not a
   third extracted helper.

## evaluation

### how well it solves the goal

solves it directly. `use.vpc.tunnel` becomes env-aware via `--env`; it reads the
per-env target from `getConfig` — the same proven contract `use.rds.capacity` ships with
(and covers with an integration test). the localhost/ssm choice is config-driven (host ==
localhost → no ssm), so all three envs flow one uniform path. and `use.rds.capacity`
collapses to a composition on top of it, so the shared logic lives in one place.

### pros

- root-cause fix (env-driven), not a symptom patch of a value map
- parity with every other operator skill → least astonishment
- inherits the already-tested failfast config contract
- removes the `--env all` over-unlock footgun
- removes the `5432` port default footgun
- **`use.rds.capacity` composes it** → deletes the duplicated config-read; one place owns
  the tunnel logic
- localhost/ssm choice is config-driven → one uniform path, no `env == test` special-case

### cons / tradeoffs

- the standalone `--bastion`/`--cluster` ergonomics go away (design stance: removed).
  any caller of the raw args must switch to `--env`. risk is bounded: no cited caller
  uses them.
- couples `use.vpc.tunnel.sh` to the consuming repo's `getConfig()` **module path**
  (`./src/utils/config/getConfig`), not just its shape — but that coupling is extant
  for `use.rds.capacity`, so it's consistent, not new.
- refactoring `use.rds.capacity` to call `use.vpc.tunnel --env` touches a skill that
  already ships with an integration test — the test must still pass (or be extended). a
  bounded, worthwhile change, but not zero-risk.

### edgecases & pit-of-success

- **env=test (or any localhost target)**: test config models `local.host = 'localhost'`
  (testdb at `localhost:7821`), with **no real bastion**. an ssm tunnel would fail.
  → handled uniformly: the skill reads config and, when host == `localhost`, skips the
    ssm tunnel and points at localhost. config decides, not a hardcoded env check.
- **absent --env** → exit 2, named.
- **invalid --env** → exit 2, "must be: test, prep, or prod".
- **placeholder `null` config** → exit 2, names each absent key (already proven by
  `use.rds.capacity.integration.test.ts`).
- **stale keyrack creds** → keyrack unlock surfaces it; no failhide.

### test coverage (comprehensive + exhaustive — no gaps)

grain: shell skill with i/o → **integration test** (`use.vpc.tunnel.integration.test.ts`),
collocated, real (no mocks), modeled exactly on `use.rds.capacity.integration.test.ts`
(same `runSkill` + `setStubConfig` harness, `given/when/then`, static aws creds to skip
keyrack, `genTempDir` with node_modules symlink).

**`use.vpc.tunnel` — every path:**

| # | case | given | expect |
|---|------|-------|--------|
| positive | localhost target (test) | config `local.host=localhost` | skips ssm; points at localhost; exit 0; no `declastruct apply` |
| positive | ssm target (prep) | config real bastion/cluster, host≠localhost | drives declastruct with `VPC_TUNNEL_*` from config; exit 0 |
| positive | ssm target (prod) | prod config | same, prod values |
| negative | absent `--env` | no arg | exit 2, `belay that`, names `--env` |
| negative | invalid `--env` | `--env bogus` | exit 2, "test, prep, or prod" |
| negative | placeholder-null bastion | config bastion=`null` | exit 2, names `database.tunnel.bastion.exid`; does not open tunnel |
| negative | placeholder-null cluster | config cluster=`null` | exit 2, names `database.tunnel.cluster.name` |
| negative | placeholder-null account | config account=`null` | exit 2, names `aws.account` |
| edge | help positional | `help` | exit 0, shows the deal, no `belay that` |
| edge | help after rhx passthrough | `--skill … --help` | exit 0 (help detected regardless of position) |
| edge | `--config` path (if retained) | explicit json | reads it instead of getConfig — only if q3 keeps `--config` |

**`use.rds.capacity` — composition regression:**
- its **extant** integration test cases (absent-env, invalid-env, placeholder-null,
  help) must **still pass unchanged** after it delegates to `use.vpc.tunnel --env`.
- add: verify it invokes `use.vpc.tunnel --env $ENV` (not the duplicated inline block)
  and still performs its own `pg_isready` wait. this proves the refactor preserves
  behavior and the delegation is real.

**snapshots:** capture the treestruct stdout (success + each `belay that`) so reviewers
vibecheck output shape and regressions surface in PRs (per `rule.require.snapshots`).

**failfast, not failhide:** absent creds/config → `ConstraintError`-style exit 2 with
hint; never a silent skip. no `it.skip`, no `expect(true).toBe(true)`.

## open questions & assumptions

### assumptions
- source of truth is `src/domain.roles/operator/skills/`; `.agent/repo=ghlitch/...`
  are currently **identical** to src (verified by read). the *sync mechanism* is
  unverified — executor must confirm whether an edit to src alone propagates to the
  `.agent` copy, or both must change together, else the installed skill goes stale.
- the consuming repo exposes `getConfig()` at the module path
  `./src/utils/config/getConfig`, returning `database.tunnel.{bastion,cluster,local}`
  and `aws.account`. **this is a path coupling, not just a shape coupling** — the
  skill `npx tsx -e "import { getConfig } from './src/utils/config/getConfig'"`
  (verified: `use.rds.capacity.sh` line 110). every consumer must place getConfig at
  exactly that path. the coupling is pre-existing (use.rds.capacity already bets on
  it), so this behavior inherits it rather than introducing it — but it is a real
  fragility: a consumer with getConfig elsewhere breaks both skills identically.
- `--env` with no default is the house standard (verified: `rule.forbid.default-env`,
  `use.rds.capacity.sh`).
- the env set is exactly `{test, prep, prod}` — a house-standard assumption
  (`prefer.env_access.prep_over_dev`), not derived from config. if a consumer adds a
  4th env, the validation list in **both** this skill and `use.rds.capacity` must
  change in lockstep.
- keyrack owner is hardcoded `ehmpath` (both skills) — deliberate per the bound brief
  `howto.keyrack` ("always use `--owner ehmpath`"), legitimate for this ehmpathy/ahbode
  role package.
- ⚠️ region is hardcoded `us-east-1` (`use.rds.capacity.sh` line 128), **not** derived
  per-env. this is an exception to the "env is the single source of truth" thesis: a
  consumer with a cluster in another region would get a cross-region mismatch — the
  same *class* of bug the wish targets. region arguably belongs in per-env config too.

## what is awkward?

- **test isn't really a "vpc tunnel".** a `test` env that points at localhost with no
  ssm tunnel, run through a skill named `use.vpc.tunnel`, is a little conceptually
  forced. but the config-driven host==localhost rule makes it *behave* consistently, so
  the leak is only in the naming, not the mechanism.
- **`--config` escape hatch vs `--env` purity.** dropping raw `--config`/`--bastion`
  loses a power-user path, but keeping it reopens the exact mismatch door the wish wants
  shut. stance is drop; `--config` returns only if a real caller needs it.
- (resolved) the "two entry points, one engine → duplicated config-read" worry is gone:
  `use.rds.capacity` composes `use.vpc.tunnel --env`, so the config-read lives in one
  skill, not two.
