# vision: eliminate failhides, ensure failfast

## the outcome world

### day-in-the-life: after

vlad runs `rhx aws.ssm.param.check --env prod --pattern 'ahbode.svc-chat.*'` with expired credentials.

```
🐈 chartin course...

🔮 aws.ssm.param.check --env prod
   └─ env: prod

   parameters that match 'ahbode.svc-chat.*':

An error occurred (ExpiredTokenException) when calling the DescribeParameters operation: The security token included in the request is expired
```

vlad immediately knows: credentials expired. the raw aws error tells the story. no guesswork. 30 seconds to fix.

### day-in-the-life: before

same command, expired credentials.

```
🐈 caught it!

🔮 aws.ssm.param.check --env prod --pattern 'ahbode.svc-chat.*'
   └─ (none found)
```

vlad thinks: "weird, i know those params exist." checks aws console. params exist. re-runs command. same result. checks profile. checks region. 15 minutes later, re-unlocks keyrack "just in case." suddenly works. "oh, credentials were expired the whole time."

### the "aha" moment

the value clicks when vlad realizes: **every minute spent debugging a failhide is a minute stolen by silent failure**. failfast gives that time back.

### before/after contrast

| before | after |
|--------|-------|
| "(none found)" — is it real? | credential error — here's the fix |
| debug spiral: 15 minutes | immediate fix: 30 seconds |
| distrust the tool | trust the tool |
| "maybe my input was wrong" | "my credentials expired" |

## user experience

### usecases

1. **credential expiry detection**: skills detect expired/invalid aws credentials and surface clear errors with hints
2. **auth failure distinction**: distinguish auth failures (exit 1, malfunction) from input errors (exit 2, constraint)
3. **skill reliability**: all skills in rhachet-roles-ghlitch failfast on errors, never swallow stderr

### contract inputs & outputs

**input**: same as today — skills take `--env`, `--pattern`, etc.

**output on error**:

```
🐈 wet paws...

{artifact} {skill-name} [--flags]
   ├─ {error-type}: {error-message}
   └─ hint: {actionable-fix}
```

**exit codes**:
- `exit 0` = success
- `exit 1` = malfunction (credential errors, aws errors, server-must-fix)
- `exit 2` = constraint (bad args, bad env, caller-must-fix)

### timeline

1. run skill with bad credentials
2. aws cli outputs error to stderr (no longer suppressed)
3. skill exits non-zero (aws cli exit code)
4. user sees raw aws error — clear what went wrong
5. user fixes credentials, retries
6. success

## mental model

### how users describe this to a friend

> "when something goes wrong, it tells you what went wrong and how to fix it. no more guessing games."

### analogies

- **smoke detector vs silent fire**: failfast is the smoke detector. failhide is the silent fire that burns your house down while you sleep.
- **check engine light**: failfast is the check engine light with a code. failhide is a car that silently loses power.

### terms

| user term | our term |
|-----------|----------|
| "it errored out" | malfunction (exit 1) |
| "i gave it bad input" | constraint (exit 2) |
| "it failed silently" | failhide (forbidden) |
| "it told me what went wrong" | failfast (required) |

## evaluation

### how well does it solve the goals?

**goal: eliminate failhides** — yes, via:
1. new rule in `.agent/repo=.this/role=any/briefs/` forbidding failhides
2. fix all extant `2>/dev/null || echo ""` patterns in skills
3. require explicit error handling for aws cli calls

**goal: failfast on auth issues** — yes, via:
- remove `2>/dev/null` — let raw aws cli errors surface naturally
- no need to parse/catch — aws cli errors are clear enough
- keyrack unlock already happens upfront — prevents most credential issues

### pros

- immediate feedback on errors
- actionable hints save debug time
- trust in tools increases
- consistent error format across all ghlitch skills

### cons

- more verbose error output (acceptable tradeoff)
- requires updating many skills (one-time cost)
- some edge cases may surface that were silently ignored (good — we want to know)

### edgecases and pit-of-success

| edgecase | pit-of-success |
|----------|----------------|
| credential expired mid-operation | detect on first aws call, fail immediately |
| partial success before auth error | report what succeeded, then fail with auth error |
| network timeout vs auth error | distinguish via error message parsing |
| rate limiting | detect and suggest backoff |

## groundwork

### external research

**aws cli error output**: verified that aws cli outputs credential errors to stderr with recognizable patterns:
- `ExpiredTokenException`
- `AccessDeniedException`
- `InvalidIdentityToken`
- `UnauthorizedAccess`

these can be captured via `2>&1` and pattern-matched.

### internal research

**extant behavior verified**:

1. **ghlitch cat persona**: `src/domain.roles/operator/briefs/im_a.ghlitch_cat.md`
   - error format: `🐈 wet paws...` for blocked
   - artifact per role: observer = 🔮, deployer = ⛵, operator = 🦺

2. **exit code semantics**: present in skill headers across all roles
   - exit 0 = success
   - exit 1 = malfunction
   - exit 2 = constraint

3. **the failhide pattern**: found in many skills
   - `aws.ssm.param.check.sh:183-186` — `2>/dev/null || echo ""`
   - `query.cloudwatch.metrics.sh:176, 272, 317`
   - `aws.cloudformation.status.sh:133`
   - and ~10 more instances

4. **correct pattern exists**: `aws.ssm.param.check.sh:138-146` shows proper credential check with `🐈 wet paws...` output — but only at skill startup, not for individual aws calls.

**vocab to reuse**:
- `wet paws` = blocked by error
- `belay that` = constraint error
- `malfunction` = exit 1
- `constraint` = exit 2
- `failhide` = forbidden pattern
- `failfast` = required pattern

**stdouts to match**: see ghlitch cat persona briefs for tree structure.

## what is awkward?

### what feels off

1. **two error phrases**: `wet paws` (malfunction) vs `belay that` (constraint) — users need to learn both. but this maps to exit 1 vs exit 2, so it's actually helpful.

2. **verbose hints**: some hints are long (e.g., `rhx keyrack unlock --owner ehmpath --env prod`). but verbosity is the tradeoff for actionability.

### where does the design fight mental model?

nowhere significant. users expect errors to be surfaced. the only adjustment is learning the exit code semantics (1 = server-must-fix, 2 = caller-must-fix).

### uncomfortable tradeoffs

1. **fixing many skills**: this is a one-time cost, but it's nonzero. we could phase it: fix `aws.ssm.param.check` first, then others in follow-up PRs.

2. **new rule may flag extant code**: the rule will be a blocker, meaning any new PR with `2>/dev/null || echo ""` will be blocked. this is intentional but may slow initial adoption. alternative: start as nitpick, upgrade to blocker after cleanup.
