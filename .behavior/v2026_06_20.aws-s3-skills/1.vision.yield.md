# vision: aws.s3.list and aws.s3.get skills for ghlitch/observer

## the outcome world

### a day-in-the-life

before: vlad debugs why a storefront won't load. he suspects the cloudfront logs might reveal the issue. he opens aws console, navigates to s3, finds the bucket, browses through nested prefixes, downloads a .gz file, gunzips it, opens in a text editor. 15 minutes gone.

after: vlad runs `rhx aws.s3.list --bucket ahbode-cloudfront-logs-prod --prefix storefront/ --since 1h` and sees the recent log files. he picks one and runs `rhx aws.s3.get --bucket ahbode-cloudfront-logs-prod --key storefront/domain=panamacityjunkremoval.com/2026-06-20-10.log.gz`. the content appears instantly, auto-gunzipped, cached locally for future reference. 30 seconds.

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| browse s3 | aws console, click through prefixes | `rhx aws.s3.list --bucket x --prefix y` |
| time filter | manual date inspection | `--since 1h` |
| view .gz file | download, gunzip, open | auto-gunzipped in terminal |
| credential handling | aws configure / sso login | keyrack pattern, same as all ghlitch skills |
| context switch | leave terminal for browser | stay in terminal |

### the "aha" moment

"wait, i can just tail s3 objects from my terminal like they're local files? and it remembers my aws creds from keyrack? turtally."

## user experience

### usecases

1. **browse s3 contents**: `rhx aws.s3.list --env prod --uri s3://logs-prod/app/`
2. **find recent objects**: `rhx aws.s3.list --env prod --uri s3://logs-prod/app/ --since 1h`
3. **view object contents**: `rhx aws.s3.get --env prod --uri s3://logs-prod/app/2026-06-20.log.gz`
4. **inspect config files**: `rhx aws.s3.get --env prod --uri s3://configs/prod/settings.json`

### contract inputs & outputs

#### aws.s3.list

```bash
# inputs (required)
--env ENV       # aws credentials env: test, prep, prod

# inputs (preferred)
--uri URI       # s3://bucket/prefix format (copy-paste friendly)

# inputs (alternative)
--bucket NAME   # s3 bucket name
--prefix PATH   # filter objects by prefix

# inputs (optional)
--since TIME    # filter by recency: 1h, 1d, 7d
--limit N       # max objects (default: 50)

# outputs
# treestruct to stdout with object list
# exit 0 = success
# exit 1 = aws malfunction
# exit 2 = constraint violation (absent args)
```

#### aws.s3.get

```bash
# inputs (required)
--env ENV       # aws credentials env: test, prep, prod

# inputs (preferred)
--uri URI       # s3://bucket/key format (copy-paste friendly)

# inputs (alternative)
--bucket NAME   # s3 bucket name
--key PATH      # object key

# outputs
# object contents to stdout
# cached to .agent/.cache/repo=ghlitch/role=observer/skill=aws.s3.get/bucket=$bucket/key=$key/asof=$now.$ext
# auto-gunzips .gz files
# exit 0 = success
# exit 1 = aws malfunction
# exit 2 = constraint violation (absent args)
```

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| quick s3 browse | yes | `aws.s3.list` with prefix filter |
| time-based filter | yes | `--since` flag |
| view compressed files | yes | auto-gunzip for .gz |
| credential reuse | yes | keyrack pattern |
| stay in terminal | yes | no browser needed |
| cache for repeat access | yes | `.agent/.cache/` pattern |

### pros

- consistent with other observer skills (query.cloudwatch.logs, etc)
- keyrack credential pattern already established
- ghlitch cat vibes output
- cache prevents redundant downloads
- auto-gunzip is a huge time saver

### cons

- bucket names must be known (no bucket discovery)
- no s3 select / query syntax (just raw fetch)
- cache can grow large if you fetch many objects

### pit of success

| edge case | how contract handles |
|-----------|---------------------|
| bucket doesn't exist | exit 1 with aws error |
| key doesn't exist | exit 1 with aws error |
| absent --bucket | exit 2 with constraint error |
| credentials not unlocked | exit 1 with keyrack hint |
| large file | no limit on size, but user sees full output |

## open questions & assumptions

### assumptions

1. users have keyrack configured with AWS_PROFILE for each env
2. s3 buckets are in the same aws account as keyrack profiles
3. users know bucket names (no discovery needed yet)
4. .gz files are gzip compressed (not other compression)

### decided questions

1. **should we support --output json** for pipe to jq?
   - **decision**: no, not for v1. use `aws s3api list-objects-v2` directly if json needed.

2. **should aws.s3.get support --lines N** to limit output?
   - **decision**: no, not for v1. pipe to `head -n N` instead.

3. **should cache have expiry or cleanup?**
   - **decision**: no, not for v1. unbounded cache is acceptable; users can delete `.agent/.cache/` manually.

4. **is ghlitch/role=observer the right home?**
   - **decision**: yes. observer's purpose is "watch all that goes on" — s3 browse fits exactly.

### external research needed

- none — s3 api is well-known, aws cli is stable

## groundwork

s3 api patterns from source skills (ahbode/app-storefronts-web):
- `aws s3 ls s3://bucket/prefix --recursive` — lists objects with date, size, key
- `aws s3 cp s3://bucket/key -` — streams to stdout, works with pipe to gunzip
- date format in s3 ls output: `2026-06-20 10:30:45` (parseable by `date -d`)

## what is awkward?

### artifact emoji mismatch

the source skills use `🦺` (operator artifact) in their output headers, but they belong in observer role which uses `🔮`. this needs an update when we pull the skills.

### cache path uses `.this`

the source aws.s3.get uses `repo=.this` in cache path, but observer skills use `repo=ghlitch`. need to update.

### no bucket discovery

users must know bucket names. a future `aws.s3.buckets` skill could list accessible buckets, but that's scope creep for this task.

### --since parse is fragile

the `--since` parse only handles `Nh`, `Nd`, `Nm` patterns. no weeks, no iso durations. probably fine for v1.
