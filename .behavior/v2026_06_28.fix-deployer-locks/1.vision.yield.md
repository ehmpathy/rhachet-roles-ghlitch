# vision — fix deployer locks 🔒⛵

> prod is sacred. by default, no one applies to prod from local.
> two per-verb meters — **`deploy.uses`** and **`provision.uses`** (one engine, separate scopes) — gate prod-mutating deployer ops. explicit allow, default block.
> independent by design: unlock provision without deploy, or deploy without provision.

---

## the outcome world

### day-in-the-life

it's a tuesday. a mechanic (or a human) needs to ship a hotfix.

```
$ rhx deploy --env prep        # ✅ just works — prep is default allow
$ rhx deploy --env prod        # 🔒 blocked — prod is sacred
```

```
🐈 wet paws...

⛵ deploy --env prod
   ├─ prod is locked: no deploy.uses granted for prod (safe default)
   └─ ask your human to grant:
        $ rhx deploy.uses set --quant 1 --env prod
```

the human, deliberately, grants a single-use prod deploy pass:

```
$ rhx deploy.uses set --quant 1 --env prod
🐈 smooth sailin!

🦺 deploy.uses set
   ├─ granted: 1
   ├─ env: prod
   └─ scope: local (this repo)
```

now the prod deploy goes through — and the meter auto-decrements to `0`, re-locking prod behind them:

```
$ rhx deploy --env prod        # ✅ allowed — consumes the grant
...
🐈 smooth sailin!

⛵ deploy --env prod
   └─ deployed to prod (deploy.uses: 1 → 0, re-locked)
```

next prod deploy is blocked again. the door swings shut on its own.

**the independence that matters:** the `deploy.uses` grant did **not** touch `provision.uses`. so you can hand someone a prod *schema migration* pass without also handing them a full *service deploy* — and vice versa:

```
$ rhx provision.uses allow --env prod    # standing access to prod migrations
$ rhx provision.terraform apply --env prod   # ✅ allowed
$ rhx deploy --env prod                  # 🔒 still blocked — deploy.uses untouched
```

### before / after

| | before | after |
|---|---|---|
| prod deploy from local | silent, one keystroke away | gated; requires explicit grant |
| prep/test deploy | one keystroke | unchanged — still one keystroke |
| "oops wrong env" | possible to nuke prod | prod refuses unless deliberately unlocked |
| org-wide prod freeze | none | `rhx deploy.uses block --global` (and/or `provision.uses`) locks every repo |
| unlock provision but not deploy | impossible | `rhx provision.uses allow --env prod` — deploy stays locked |

### the "aha"

prod isn't protected by *remembering to be careful*. it's protected by a **default-closed door** that you must consciously open, that **closes itself** after one use. the safety is structural, not behavioral — a pit of success.

## user experience

### usecases & goals

(below, `<meter>` is `deploy.uses` **or** `provision.uses` — same grammar for both.)

| who | goal | how |
|---|---|---|
| mechanic | ship to prep freely | `rhx deploy --env prep` (no gate) |
| mechanic | attempt prod | blocked with clear instructions to escalate |
| human | grant a one-shot prod deploy | `rhx deploy.uses set --quant 1 --env prod` |
| human | grant standing prod migrations | `rhx provision.uses allow --env prod` |
| human | unlock provision but not deploy | `rhx provision.uses allow --env prod` (deploy stays locked) |
| human | re-lock | `rhx <meter> block --env prod` / `del` |
| human | freeze prod everywhere | `rhx <meter> block --global` |
| human | set org-wide policy | `rhx <meter> block --org @all` |
| anyone | check current lock state | `rhx <meter> get` |

### contract — inputs & outputs

**two consumer-facing skills, one shared engine** (role=deployer, artifact 🦺): `deploy.uses` and `provision.uses` are thin wrappers over the same `*.uses` engine, parameterized by meter name. each is **env-keyed** — grants target a specific `--env` (prod is the one that matters; prep/test default-allow).

```
rhx <meter> get                              # show resolved state across all scopes & envs
rhx <meter> set --quant <N> --env prod       # grant N prod uses (local)
rhx <meter> allow          --env prod        # grant standing/unlimited (local)
rhx <meter> block          --env prod        # revoke (local)  [alias: del]
rhx <meter> block  --global                  # circuit-break this verb's prod, everywhere
rhx <meter> allow  --global                  # lift global lock
rhx <meter> allow  --org ehmpathy --env prod # org-level policy
rhx <meter> block  --org @all                # org default
```

where `<meter>` ∈ { `deploy.uses`, `provision.uses` }.

guarantee:
- exit 0 = state read/set
- exit 2 = constraint (only humans may grant; absent args)

each consumer skill consults **its own** meter before any prod mutation:
- `deploy`, `aws.cloudformation.rollback` → consult **`deploy.uses`**
- `provision.terraform`, `provision.database` → consult **`provision.uses`**

on block → exit 2 with escalation hint. on allow → proceed; if quota-based, decrement + auto-revoke.

**under the hood:** one engine (the `*.uses` resolution cascade + storage + TTY guard, mirroring `git.commit.operations.sh`) takes a `--meter` parameter. `deploy.uses` and `provision.uses` are the two named entry points; they share all logic and differ only in which `.meter/<name>.jsonc` they read/write.

### timeline

```
human grants  →  mechanic applies to prod  →  grant auto-consumed  →  prod re-locked
   (deliberate)        (one op)                  (structural)           (default-closed)
```

## mental model

**how a user describes it to a friend:**
> "prod is locked by default. you can't deploy to prod from your laptop unless someone unlocks it — and it re-locks itself after one push. like a self-closing vault door."

**analogies:**
- a **self-closing vault door** — opens on deliberate grant, swings shut after you pass
- a **breaker panel** — global block is the master switch; org/local are sub-breakers
- the **commit meter you already know** (`git.commit.uses`) — same shape, applied to prod deploys instead of commits

**terms — theirs vs ours:**

| they say | we say |
|---|---|
| "unlock prod deploys" | grant `deploy.uses --env prod` |
| "unlock prod migrations" | grant `provision.uses --env prod` |
| "lock it" | `block` / `del` |
| "freeze all prod" | `<meter> block --global` |
| "one-time pass" | `set --quant 1` (single-use grant, auto-revokes) |

reuses the established `*.uses` vocabulary and the global > org > local scope model — so anyone who knows `git.commit.uses` already knows this. the only new idea is **two named meters over one engine**, so deploy and provision lock independently.

## evaluation

### how well it solves the goals

| wish goal | met? | how |
|---|---|---|
| repo-grain prod gate | ✅ | local `.meter/deploy.uses.jsonc` + `.meter/provision.uses.jsonc` |
| explicit allow, default block on prod | ✅ | unset = blocked (safe default) |
| prep/test default allow | ✅ | meter consulted only when `--env prod` |
| org + global scope | ✅ | mirrors `git.commit.uses` precedence |
| gate all deployer prod ops | ✅ | gate inserted in every prod-mutator skill |
| name the skills | ✅ | `deploy.uses` + `provision.uses` (per-verb, one shared engine) |
| independent deploy vs provision control | ✅ | two meters — unlock one without the other |

### pros

- structural safety: default-closed, self-relocking
- zero friction for prep/test (the 95% case)
- familiar: reuses `*.uses` mental model & scope precedence
- global circuit-breaker for org-wide freezes
- consistent ghlitch cat ergonomics (🦺 artifact, tree output)

### cons / tradeoffs

- new ceremony for legit prod deploys (mitigated: one-shot grant)
- two `*.uses` meters to maintain — but they share one engine, so the cost is one implementation + two thin wrappers
- quota auto-revoke could surprise someone mid-multi-step prod rollout (e.g. terraform apply then a follow-up) — `allow --env prod` standing grant covers it
- two meters = two places to check/unlock — a touch more to reason about than one (the independence payoff is worth it)

### edgecases & pit-of-success

| edgecase | behavior |
|---|---|
| `--env prod` with no grant | blocked, exit 2, escalation hint |
| `--env prep/test` | never consults meter — always allowed |
| `provision plan --env prod` (no grant) | **runs ungated** — plan previews safely; only apply is gated |
| `provision apply --env prod` (no grant) | blocked, exit 2, escalation hint |
| `<meter> get` with no grant set | reports "prod locked (default)" |
| `provision.uses` allowed, `deploy.uses` not | provision proceeds; deploy still blocked (independence) |
| global block + local allow | **global wins** — blocked (breaker model) |
| mechanic tries to `<meter> set` | refused — only humans may grant (TTY guard) |
| corrupt meter file | treated as blocked (fail-closed) |
| read-only prod op (`cloudformation.status`) | **not gated** — no mutation, no lock |

## threat model & scope boundary

what the meter actually guarantees — stated plainly so no one mistakes it for an airtight prod lock:

- **vs an autonomous agent (mechanic): a hard wall.** the `set` grant is TTY/human-guarded — an agent cannot self-grant prod access. this is the primary protection.
- **vs a human: a deliberate-friction speed-bump + an external freeze.** a human can self-grant `--quant 1`, so for them the meter converts a reflexive one-keystroke prod apply into a conscious two-step act. the real hard-stop for humans is the **global/org freeze**, controlled by someone other than the actor.

**scope boundary:** the gate lives in the `rhx` skills, so it protects the **rhx-skill path** (`rhx deploy --env prod`). it does **not** guard raw `npm run deploy:prod` or `terraform apply` run directly — those tool-level invocations are out of scope by design (CI relies on them). the meter secures the ergonomic default path, not every conceivable path to prod.

**package-wide enforcement:** these skills are defined in `rhachet-roles-ghlitch` and consumed by *every* repo that uses the ghlitch deployer role. the prod gate therefore enforces in **all** consumer repos, not only this one — `default-block prod` becomes a fleet-wide default. **wisher confirmed this is intended** ("prod sacred everywhere"), same as `git.commit.uses`.

**org-scope dependency:** org resolution reads `.agent/keyrack.yml#org` (as `git.commit.uses` does). a consumer repo that omits that field cannot derive org policy; per the reference's fail-closed posture, that yields **blocked** for prod — safe, but possibly surprising in a repo that never configured org. (this repo has `org: ehmpathy`, `keyrack.yml:1`.)

## open questions & assumptions

### assumptions made

1. **two per-verb meters** (wisher decided): `deploy.uses` and `provision.uses`, role=deployer, repo=ghlitch, artifact 🦺. they share **one engine** parameterized by meter name — so deploy and provision lock independently ("sometimes allow provision but not deploy, and vice versa").
2. **storage namespace** mirrors the pattern, one file per meter:
   - global: `~/.rhachet/storage/repo=ghlitch/role=deployer/.meter/{deploy,provision}.uses.global.jsonc`
   - org: `~/.rhachet/storage/repo=ghlitch/role=deployer/.meter/{deploy,provision}.uses.org.jsonc`
   - local: `<repo>/.meter/{deploy,provision}.uses.jsonc` (env-keyed inside)
3. **only `--env prod` is gated.** non-prod envs bypass the meter entirely (test where the skill accepts it — note `deploy` and `provision.database` accept only `prep|prod`; `provision.terraform` also accepts `test`).
4. **both grant forms supported** (mirrors `git.commit.uses`): `set --quant N` for a one-shot self-revoking pass, `allow` for standing access, `block`/`del` to revoke. the engine handles all three.
5. **only prod-mutating ops are gated** (see triage below).

### plan stays open, apply stays gated

a first-class requirement: **`plan` against prod must still run; only `apply` is gated.** plan is how you preview a prod change safely — a lock on it would punish the careful path. each skill separates the two cleanly:

| skill | how plan vs apply is told apart | gate fires when |
|---|---|---|
| `provision.database` | explicit `--mode plan\|apply` flag (`:139-155`) | `env==prod && mode==apply` |
| `provision.terraform` | the terraform subcommand in `TERRAFORM_ARGS` (`:241`) | `env==prod && subcommand is a writer` |

for terraform, classify by a **read-only allowlist** — `plan`, `validate`, `show`, `output`, `fmt` pass through ungated; all other subcommands (`apply`, `destroy`, `import`, `state rm/mv`, `taint`, …) are gated. fail-closed: an unrecognized subcommand counts as a writer, so a new mutation verb cannot slip past.

**the RDS-wake nuance (a refinement of "gate before the first prod touch").** `provision.database` wakes prod RDS via `use.rds.capacity:168` *before* either plan or apply runs — because **plan also needs the DB awake** to diff the live schema. so the gate must sit at the **apply**, not before the wake. the wake is a benign capacity action that auto-re-pauses, and is the necessary cost of an allowed plan; the schema mutation is the sacred act, and that is what `provision.uses` guards. (this revises the earlier "gate before any prod side-effect" stance — that would have blocked plan, which we explicitly want open.)

**execution note — within the apply path, gate before the mutation runs.** once a prod apply is established, the gate check must precede the actual `npm run provision:schema:apply` / `terraform apply` call, so a blocked attempt never executes the write.

## what is awkward?

1. **two `*.uses` families, different rules.** `git.commit.uses` and `radio.uses` both default *closed* and gate their op unconditionally. ours is **selectively closed** — only `prod`. test/prep skip it. that asymmetry ("the meter only exists for one env value") is a slight conceptual wrinkle vs the reference meters.

2. **quota semantics on a multi-step prod rollout.** since plan is ungated, a single `--quant 1` covers the one *apply* — the common case — without nag. the residual wrinkle: a rollout that needs *several* applies in a row (e.g. terraform `apply` then a corrective `apply`) consumes a grant each time. the `allow --env prod` standing grant covers that, but then it's no longer "default sacred" until they remember to re-block.

3. **"prod is sacred" vs cloudformation.rollback.** rollback is often an *emergency* prod action — exactly when you don't want a locked door. a gate on it is correct for safety but could add friction during an incident. **decided: gate it under `deploy.uses`** (a grant is one command away).

4. **where the gate logic lives.** to avoid copy-paste of the resolution cascade into 4 skills, the engine should live in one shared sourced file (e.g. `uses.operations.sh`) that all skills source — mirroring `git.commit.operations.sh`. the two meters (`deploy.uses`, `provision.uses`) and the four gated skills all call it with a meter-name parameter. a small architectural commitment to surface now.

5. **two meters, one engine — keep them truly DRY.** the risk is divergence: a fix to `deploy.uses` that doesn't reach `provision.uses`. the engine-with-parameter design avoids this by construction (one code path, two names), but it must be honored in execution — no copy-paste forks.
