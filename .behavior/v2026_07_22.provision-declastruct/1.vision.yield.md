# 1.vision — provision.declastruct as a deployer skill

## .what (the wish, distilled)

move the `provision.declastruct` skill out of the catch-all `.agent/repo=.this/role=any/skills/`
and into the deployer role at `.agent/repo=ghlitch/role=deployer/skills/`, and while we move
it, bring it up to the **latest best practices** set by its two shipmates:

- `provision.terraform` (⛵ deployer)
- `provision.database` (⛵ deployer)

the skill's job stays the same: **declaratively provision remote infrastructure via
declastruct** — plan the diff, then apply it. what changes is the *ergonomics*: the ghlitch cat
mascot, treestruct output, `--env` selection, semantic exit codes, a prod write-gate, and a
forward-contract stdout that CI can grep.

---

## the outcome world

### day-in-the-life

a deployer (human or clone) wants to provision the demo aws lambdas that the observer's
CloudWatch tests depend on. today they either:

- reach into `.agent/repo=.this/role=any/skills/provision.declastruct.sh` (a role-less orphan), or
- copy the raw `npx declastruct plan --wish … --into … && npx declastruct apply --plan …`
  incantation out of the account readme.

**after** this wish, they run the same shape of command they already run for terraform and the
database:

```sh
# preview the diff
rhx provision.declastruct --wish provision/aws.infra/account=demo/resources.ts --env test --mode plan

# apply the reviewed plan
rhx provision.declastruct --wish provision/aws.infra/account=demo/resources.ts --env test --mode apply
```

and they get the same cat-and-boat output they get from the other two:

```
🐈 chartin course...

⛵ provision.declastruct --wish provision/aws.infra/account=demo/resources.ts --env test --mode plan
   ├─ wish: provision/aws.infra/account=demo/resources.ts
   ├─ env: test
   ├─ mode: plan
   └─ plan infra changes...

   … declastruct plan stdout flows through unmodified …

🐈 smooth sailin!

⛵ provision.declastruct --wish … --env test --mode plan
   └─ planned → provision/aws.infra/account=demo/resources.ts.plan.json
```

### before / after contrast

| dimension | before (orphan skill) | after (deployer skill) |
|-----------|----------------------|------------------------|
| home | `repo=.this/role=any/skills/` | `repo=ghlitch/role=deployer/skills/` |
| discoverability | not in the deployer readme's skill table | listed alongside `provision.terraform`, `provision.database` |
| voice | bare `echo "plan …"` | 🐈 mascot + ⛵ treestruct |
| env | hardcoded `keyrack unlock --env test` | explicit `--env` (no default — pit of success) |
| exit codes | `exit 2` for everything, no semantics doc | 0 = ok, 1 = malfunction, 2 = constraint |
| help | plain text | `🐈 heres the deal...` help block |
| prod safety | none (test only) | prod write-gate via `provision.uses` + `--auth as-cicd` |
| CI parity | drifts from `.declastruct.yml` | same `--into <wish>.plan.json` handoff CI already uses |

### the "aha" moment

the value clicks when a deployer realizes **all three provisioners now rhyme**. one mental
model — `rhx provision.<thing> … --env <env> --mode <plan|apply>` — covers terraform, the
database, and now declastruct. the orphan that lived off in `role=any` is now a first-class
citizen of the deployer's toolbelt, and it *sounds* like its shipmates.

## user experience

### usecases & goals

- **provision demo aws infra** — stand up the lambdas + iam role that the observer's
  CloudWatch integration tests query (the primary current usecase; see
  `provision/aws.infra/account=demo/readme.md`).
- **provision github repo config** — the second extant wish
  (`provision/github.repo/resources.ts`).
- **any future declastruct wish** — the contract is wish-file-agnostic, so a new
  `provision/**/resources.ts` needs no new skill.

### contract inputs & outputs

**inputs:**

| flag | required | values | sense |
|------|----------|--------|-------|
| `--wish` | yes | path to a `resources.ts` | the declastruct declaration to provision |
| `--env` | yes | `test` \| `prep` \| `prod` | which environment's credentials to unlock; also exported so the wish file may reuse it |
| `--mode` | yes | `plan` \| `apply` | preview the diff, or apply the reviewed plan |
| `--auth` | no | `as-cicd` | prod-apply auth: defer the prod gate to the github-environment approval in CI |
| `<extra args>` | no | any | any arg the skill does not consume is forwarded verbatim to the `declastruct` invocation |
| `help` / `--help` / `-h` | — | — | usage |

**`--env` is always required and always consumed by the skill** — it drives `keyrack unlock`
(the wish file's `keyrack.source({ mode: 'lenient' })` no-ops without an unlocked session, and a
wish file does not always name an env itself). the skill **exports** the chosen env into the
process environment ("autoset"), so a wish file may *reuse* the given env instead of a hard-coded
one — or it may still source its own env separately. both are valid; the wish file decides.

**arg pass-through**: as `provision.terraform` does (it forwards its `TERRAFORM_ARGS`), any
argument the skill does not itself consume (`--wish`, `--env`, `--mode`, `--auth`) is collected
and appended to the `npx declastruct plan|apply …` call. this keeps the skill a thin, transparent
wrapper — a new declastruct flag propagates without a skill change.

**outputs:**

- **stdout** — the declastruct plan/apply output flows through **unmodified** (forward
  contract), framed by the 🐈/⛵ headers. a caller can `| tee ./plan.log` and grep
  `"Everything is in sync"` exactly like `.declastruct.yml` does.
- **plan file** — `plan` mode writes `<wish>.plan.json`; `apply` mode reads it back, so we
  apply exactly the plan that was reviewed (gitops safety, matches CI).
- **exit code** — `0` provisioned, `1` malfunction (declastruct/aws error), `2` constraint
  (absent/bad args).

### timeline (plan → apply)

```
deployer                     provision.declastruct                 declastruct / aws
   │  --mode plan                    │                                    │
   ├────────────────────────────────►│  unlock keyrack --env <env>        │
   │                                 ├───────────────────────────────────►│  read remote state
   │                                 │  npx declastruct plan --into <wish>.plan.json
   │◄─── plan stdout + plan.json ────┤                                    │
   │  (review the diff)              │                                    │
   │  --mode apply                   │                                    │
   ├────────────────────────────────►│  [prod? → provision.uses gate]     │
   │                                 ├───────────────────────────────────►│  apply <wish>.plan.json
   │◄─── apply stdout, exit 0 ───────┤                                    │
```

## mental model

### how a user describes it to a friend

> "it's the boat skill for declastruct — same as `provision.terraform` but for our
> declarative aws/github resources. you `--mode plan` to see the diff, `--mode apply` to ship
> it. the cat tells you if you're sailing smooth or got wet paws."

### analogies / metaphors

- **plan/apply** = the terraform mental model users already hold: preview, then commit.
- **the reviewed-plan handoff** = "sign the blueprint, then build exactly that blueprint" —
  apply consumes the plan file plan produced, so no surprise re-diff at apply time.
- **the cat + boat** = the ghlitch deployer house style; the skill *joins the crew*.

### their terms vs our terms

| they say | we say |
|----------|--------|
| "the resources file" / "the wish" | `--wish` (a declastruct declaration / `resources.ts`) |
| "dry run" / "preview" | `--mode plan` (never the forbidden dry-run term) |
| "ship it" / "deploy" | `--mode apply` |
| "the plan" | `<wish>.plan.json` (the serialized diff) |
| "infra" | declared resources (aws iam/lambda, github repo) |

## evaluation

### how well it solves the goals

- **fully** solves "move to deployer role" — a pure relocation + rename of home.
- **fully** solves "conform to best practices" — mascot, treestruct, `--env`, exit codes,
  help block, prod gate, forward-contract stdout all mirror the two shipmates.
- **preserves** the extant plan→apply→file handoff, which is *more* faithful to CI than
  provision.database's independent plan/apply.

### pros

- one consistent provisioner mental model across terraform / database / declastruct.
- discoverable (deployer readme table) and voiced (ghlitch cat).
- prod writes gated like the database's prod writes — no accidental prod infra apply.
- CI (`.declastruct.yml`) and the local skill converge on the same `--into <wish>.plan.json`.

### cons / tradeoffs

- **`--env` is partly redundant with the wish file.** `resources.ts` already sources its own
  credentials (`keyrack.source({ env: 'test' })`, line 17). so `--env` on the skill unlocks a
  keyrack session, but the wish file may still self-select `test`. we must reconcile this (see
  awkward + open questions) rather than let two env sources silently disagree.
- **`--env prep|prod` are aspirational today.** only `test` is exercised by a live wish
  (`account=demo` sources `env: 'test'`); no prep/prod declastruct wish exists yet. the full enum
  matches the shipmates and looks forward, but a reader should not assume prep/prod declastruct
  provision runs out of the box until a wish sources those envs.
- **the prod gate guards a forward path.** the current only-usecase is test-env demo infra, so the
  prod gate protects a path no live wish walks yet. the wisher chose full conformance over deferral:
  the gate is cheap, and its absence would be a foot-gun the moment a prod wish appears.

### edgecases & pit-of-success

| edgecase | how the contract keeps users safe |
|----------|-----------------------------------|
| absent `--wish` | exit 2, "absent required arg: --wish" |
| `--wish` path missing on disk | exit 2, "wish not found: <path>" |
| absent / bad `--env` | exit 2, no default env (never silently hit prod) |
| bad `--mode` | exit 2, "must be: plan or apply" |
| `apply` with no prior plan file | exit 2, "run --mode plan first" (matches extant behavior) |
| prod apply, local, ungated | blocked by `provision.uses` meter until a human grants quota |
| prod apply in CI | `--auth as-cicd` defers to github-environment approval (matches provision.database) |
| declastruct/aws error | exit 1 (malfunction), stderr surfaces the real error (no failhide) |

## open questions & assumptions

### assumptions made

1. **the skill stays wish-file-agnostic** (`--wish <path>`), not `--which <alias>` like
   provision.database. declastruct wishes live at arbitrary `provision/**/resources.ts` paths,
   so a path arg fits better than a fixed alias set.
2. **plan→apply file handoff is retained** — apply consumes the plan file plan wrote, for
   gitops fidelity and CI parity.
3. **`--env` is added for best-practice conformance and to drive `keyrack unlock`**, even
   though the wish file also sources credentials.
4. the move is a **relocation, not a rewrite** — same declastruct CLI calls
   (`npx declastruct plan --wish … --into …`, `npx declastruct apply --plan …`), verified
   against `.github/workflows/.declastruct.yml` (lines 68, 139) and the account readme.

### questions, triaged

1. **[answered — wisher] env reconciliation**: `--env` is **required** and always consumed by the
   skill for `keyrack unlock` (a wish file does not always name an env, so the skill must take it).
   the skill **exports** the chosen env ("autoset") into the process environment, so the wish file
   may *reuse* the given env — or source its own separately. both are valid; the wish file chooses.
   there is **no mismatch to surface** — the wisher settled that the skill owns `--env` for keyrack
   and the wish file is free to reuse or ignore it. **decided.**
2. **[answered — wisher] prod gate scope**: add the **full** `provision.uses` prod write-gate +
   `--auth as-cicd` now, at full best-practice parity with provision.database. gate fail-closed:
   `plan` (read) stays open, every non-plan mode (write) is gated. the gate reuses
   `uses._.check.sh`, and `--auth as-cicd` defers the prod-write gate to the github-environment
   approval in CI. the wisher confirmed conformance to the shipmate over deferral. **decided: add
   the full gate.**
3. **[answered] plan-file location**: switch to CI's `<wish-path>.plan.json` convention (drop the
   extant `<wishdir>/.temp/plan.json`). answerable now by logic: matching CI's exact path removes
   a drift seam between the skill and `.declastruct.yml`, at no cost. **decided: converge on
   `<wish-path>.plan.json`.**
4. **[answered] fate of the old file**: "move" means delete the source, so
   `.agent/repo=.this/role=any/skills/provision.declastruct.sh` is deleted (not copied), and
   `provision/aws.infra/account=demo/readme.md` is updated to reference the new skill instead of
   raw `npx declastruct` calls. answerable now: "move" is unambiguous, and a stale readme pointing
   at raw commands beside a nicer skill is a defect. **decided: delete old file + update readme.**

### must-research-externally

- **[research] none beyond what's below** — declastruct is a local devDependency, no live external
  API. the one execution-time verification flagged: run `npx declastruct --help` at build time to
  confirm flag spelling beyond the three corroborating usages (not a vision gap).

## what is awkward?

1. **two sources of env truth — RESOLVED by the wisher.** `--env` on the skill *and*
   `keyrack.source({ env })` inside the wish file both can name an environment. the wisher settled
   the model: the skill **always takes `--env`** (required, for keyrack) and **exports** it so the
   wish file may *reuse* the given env or source its own separately — the wish file's choice, with
   no mismatch error. so the apparent friction is by-design flexibility, not ambiguity: the skill
   owns the keyrack env; the wish file opts in to reuse. this removes what was the #1 open question.

2. **`--which` vs `--wish` asymmetry.** provision.database uses `--which livedb` (a fixed
   alias); declastruct uses `--wish <path>` (an arbitrary file). the trio won't be perfectly
   uniform. this feels acceptable — declastruct genuinely has many wish files, the database has
   one target — but it's a visible seam in the "they all rhyme" story.

3. **a prod gate with no prod usecase yet — RESOLVED by the wisher.** the full `provision.uses`
   gate guards a path no live wish walks (the only wish is test-env demo infra) — the wet-over-dry
   tension. the wisher chose full conformance with provision.database: add the gate now. the cost
   is low (~5 lines, reuse `uses._.check.sh`) and the safety is real, so the tradeoff resolves
   toward conformance. named here for the record, but no longer open.

4. **plan-file convention divergence.** the extant skill writes `<wishdir>/.temp/plan.json`;
   CI writes `<wish-path>.plan.json`. keeping two conventions is drift; converging is a small
   behavior change to the moved skill. minor, but a decision to make explicitly.
