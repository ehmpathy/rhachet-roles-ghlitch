# vision: promote generic skills from svc-jobs

## the outcome world

### day-in-the-life: after

vlad is debugging a production issue. instead of hunting for the right incantation of aws cli commands, remembering which keyrack keys to unlock, and manually setting up vpc tunnels, vlad just runs:

```sh
# check recent errors in logs
rhx query.cloudwatch.logs --env prod --filter ERROR --since 1h

# query the database to verify state
echo "SELECT * FROM jobs WHERE status = 'STUCK' LIMIT 10" | rhx query.database --env prod --sql @stdin

# invoke a command to unstick them
rhx invoke.command --env prod --name unstickJobs --job-ids 123,456,789
```

each skill handles credentials, vpc tunnels, readonly enforcement, and cleanup. vlad focuses on the investigation, not the plumbing.

### day-in-the-life: before

vlad needs to check prod logs:
1. open aws console
2. find the right log group
3. figure out logs insights syntax
4. copy results somewhere useful

vlad needs to query the database:
1. remember which bastion to use
2. set up ssm tunnel manually
3. hope they remembered to make it readonly
4. run psql, copy results
5. clean up tunnel

vlad needs to invoke a command:
1. ssh into something? deploy a lambda? curl an endpoint?
2. hope the right env vars are set
3. hope they don't accidentally hit the wrong environment

### the "aha" moment

> "i just ran three commands and solved a prod issue without opening a browser, without setting up tunnels, without worrying about accidentally writing to the database. the tools handled all of that for me."

## user experience

### usecases and goals

| usecase | goal | skills used |
|---------|------|-------------|
| investigate production incident | understand what's happening | `query.cloudwatch.logs`, `query.cloudwatch.metrics`, `query.database` |
| verify data state | check if records are correct | `query.database` |

### contract inputs & outputs

#### query.cloudwatch.logs

```sh
# inputs
rhx query.cloudwatch.logs \
  --env prod \                    # required: prep, prod
  --lambda handleJobCreate \      # optional: specific lambda
  --filter ERROR \                # optional: filter terms (repeatable)
  --since 1h \                    # optional: time range (default: 1h)
  --limit 100 \                   # optional: max events (default: 100)
  --tail                          # optional: follow in real-time

# outputs
# - logs to stdout
# - .agent/.cache/query.cloudwatch.logs/query.output.md (summary with stats)
```

#### query.cloudwatch.metrics

```sh
# inputs
rhx query.cloudwatch.metrics \
  --env prod \                    # required: prep, prod
  --namespace lambda \            # optional: lambda (default) or sqs
  --metric Errors \               # optional: metric name
  --since 7d \                    # optional: time range (default: 7d)
  --lambda handleJobCreate        # optional: filter to specific lambda

# outputs
# - sorted table to stdout (highest count first)
```

#### query.database

```sh
# inputs
echo "SELECT * FROM jobs WHERE status = 'STUCK'" | \
rhx query.database \
  --env prod \                    # required: test, prep, prod
  --sql @stdin \                  # required: sql query or @stdin
  --format table \                # optional: table (default), csv, json
  --limit 100                     # optional: max rows (default: 100, max: 1000)

# outputs
# - query results to stdout in specified format

# safety guarantees
# - connection-level readonly enforcement
# - transaction-level readonly enforcement
# - postgresql rejects INSERT/UPDATE/DELETE/DROP at both layers
```

### timeline

typical investigation flow:

```
t+0m   rhx query.cloudwatch.metrics --env prod --metric Errors --since 1h
       → see spike in handleJobCreate errors

t+1m   rhx query.cloudwatch.logs --env prod --lambda handleJobCreate --filter ERROR --since 1h
       → see "deadlock detected" errors

t+3m   echo "SELECT * FROM jobs WHERE status = 'PROCESSING' AND updated_at < NOW() - INTERVAL '1 hour'" | \
       rhx query.database --env prod --sql @stdin
       → see 47 stuck jobs

t+5m   rhx invoke.command --env prod --name unstickJobs --older-than 1h
       → jobs requeued

t+6m   rhx query.cloudwatch.metrics --env prod --metric Errors --since 10m
       → error rate dropping, incident resolved
```

## evaluation

### how well does it solve the goals?

| goal | how addressed |
|------|---------------|
| no browser switching | all terminal-native |
| no tunnel management | `use.rds.capacity` handles it automatically |
| no credential juggling | `keyrack unlock` integrated |
| no accidental writes | readonly enforcement at connection + transaction level |
| no wrong environment | explicit `--env` required |

### pros

- **pit of success**: hard to accidentally write to prod db
- **cognitive offload**: don't need to remember aws cli syntax
- **composable**: skills build on each other (`invoke.command` calls `use.rds.capacity` calls `use.vpc.tunnel`)
- **consistent**: same patterns across observer/supporter/deployer
- **cacheable**: log queries output to `.agent/.cache/` for reference

### cons

- **keyrack dependency**: must have keyrack set up and unlocked
- **vpc tunnel complexity**: `use.vpc.tunnel` needs sudo for `/etc/hosts`
- **service-specific**: some skills assume service conventions (`src/contract/commands/`)

### edgecases and pit-of-success

| edgecase | mitigation |
|----------|------------|
| user forgets `--env` | required flag, fails fast |
| user tries to write via query.database | readonly at connection + transaction level |
| user tries to run command that doesn't exist | `--list` shows available, clear error otherwise |
| vpc tunnel times out | 180s timeout with polling, clear error if db never wakes |
| user runs against wrong env | explicit `--env` flag, no defaulting |

## open questions & assumptions

### assumptions made

1. **keyrack is available**: skills depend on `rhx keyrack unlock` for credentials
2. **aws ssm access**: vpc tunnels require ssm session manager
3. **service conventions**: `invoke.command` assumes `src/contract/commands/` structure
4. **readonly is sufficient**: for query.database, we assume readonly queries cover investigation needs

### questions answered via research

1. **support skills visibility** [answered]: `use.rds.capacity` and `use.vpc.tunnel` WILL be exposed as standalone skills
   - evidence: in svc-jobs, `use.rds.capacity.sh` IS registered in `boot.yml` as user-faced
   - evidence: also called internally by invoke.command, invoke.vital
   - rationale: user may want manual tunnel for psql access; idempotent so double-call safe

2. **cloudformation skills** [answered]: aws.cloudformation.status and aws.cloudformation.rollback WILL be included in deployer
   - evidence: aws.cloudformation.status diagnoses deploy failures via stack state
   - evidence: aws.cloudformation.rollback continues stuck rollbacks
   - rationale: wisher said "probably some other ones" — these complete the deploy workflow

### design decisions [answered]

1. **invoke.vital in observer** [answered]: the wish explicitly places invoke.command in supporter but is silent on invoke.vital. vision places invoke.vital in observer since vitals are observational (health checks). vitals that act should be commands instead.
   - rationale: separation of read vs write aligns with observer vs supporter roles

2. **skill names use query.*** [answered]: the wish uses `query.database`, `query.cloudwatch.logs`, `query.cloudwatch.metrics` — so we follow that, not the source's `show.*` names.
   - rationale: wisher is explicit, no ambiguity

### questions to research externally

none — no external dependencies beyond aws (already known to work)

## what is awkward?

### sudo requirement for use.vpc.tunnel

`use.vpc.tunnel` modifies `/etc/hosts` which requires sudo. this could surprise users or fail in restricted environments.

**mitigation options:**
1. document clearly
2. fail fast with helpful error if sudo unavailable
3. explore alternatives (ssh config, etc.)

### service-specific conventions

`invoke.command` assumes `src/contract/commands/` structure. this is fine for ehmpathy services but may not generalize.

**mitigation**: document clearly that this skill works with ehmpathy service conventions

### skills require generalization (not direct copy)

**discovered during self-review**: skills in svc-jobs have service-specific dependencies:

1. `getDatabaseConnection` — imported from `src/utils/database/`, not a shared package
2. `--owner ehmpath` — hardcoded keyrack owner
3. `getConfig()` — imported from service code for database credentials

**implication**: promotion is not just "copy skills" — it requires:
- extracting service-specific imports to shared packages (or parameterize)
- deriving keyrack owner from repo context (not hardcode)
- standardizing getDatabaseConnection pattern

**mitigation**: scope includes generalization work, not just copy-paste

### keyrack as prerequisite

all skills depend on keyrack being set up. first-time users hit friction.

**mitigation**: clear error messages with setup instructions

### invoke.vital between observer and supporter

some vitals are read-only (check health), others can act (repair). unclear where to place.

**decision [answered]**: place in observer. vitals that act should be commands instead. documented in design decisions section.
