- src/domain.roles/deployer/skills/provision.database.sh
- src/domain.roles/deployer/skills/provision.terraform.sh

The codepath trees in the blueprint describe multiple .sh skill files (e.g. query.cloudwatch.logs.sh, aws.cloudformation.rollback.sh, use.testdb.sh, provision.terraform.sh) as long sequences of inline steps. These mix transformer-style compute (parse args, validate env, format output, check stack state) with communicator-style i/o (unlock keyrack, export aws credentials, run aws cli, docker run, terraform plan/apply) inside single operations. This matches the blocker pattern of unclear grain and orchestrator with raw i/o inline. The .sh files are not decomposed into reusable leaf operations; instead they contain the logic directly, preventing narrative reading, recomposition, and clear boundaries. Only a few skills (query.database.ts, use.vpc.tunnel.ts) are shown decomposed to .ts orchestrators that call named transformers/communicators; the rest are monolithic.

**snippet**:
```text
query.cloudwatch.logs.sh
├── [+] parse args (--env, --lambda, --filter, --since, --limit, --tail)
├── [+] validate env (prep | prod)
├── [+] unlock keyrack --owner ehmpath
├── [+] export aws credentials
├── [+] run aws logs filter-events
├── [+] format output
└── [+] write cache to .agent/.cache/
```

# blocker.2: Inline env validation planned as duplicated logic instead of transformer

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.prefer.decomposable-architecture.md

**locations**:
- .behavior/v2026_05_12.operator/3.3.1.blueprint.product.yield.md

The generalization requirements section explicitly shows env validation as inline shell code to be copied into each .sh script. This is compute logic (transformer grain) that should be extracted to a reusable named operation like validateEnv({ env }) for explicit inputs/outputs and recomposition. Duplicating the regex check across many skills creates the exact problem the rule warns against: poor decomposition compounds over time. The blueprint should plan for the validation to be a leaf transformer called from the skill orchestrators.

**snippet**:
```bash
if [[ ! "$ENV" =~ ^(test|prep|prod)$ ]]; then
  echo "error: --env must be test, prep, or prod" >&2
  exit 1
fi
```

# blocker.3: Unclear grain for shell skill entry points

- src/domain.roles/deployer/skills/aws.cloudformation.rollback.sh
- src/domain.roles/deployer/skills/provision.terraform.sh

Most .sh files are shown only via imperative step lists with no grain label (unlike the explicit "(orchestrator — composes transformers + communicators)" labels on the two .ts files). The .sh descriptions include both orchestration (calling other things) and leaf work (parsing, aws calls, docker, terraform) without splitting into communicators for i/o boundaries and transformers for compute. This creates the unclear grain blocker pattern where it is ambiguous whether the script is a thin orchestrator, a communicator, or a mixed operation.

**snippet**:
```text
aws.cloudformation.rollback.sh
├── [+] parse args (--env)
├── [+] validate env (prep | prod)
├── [+] check stack state = UPDATE_ROLLBACK_FAILED
├── [+] unlock keyrack --owner ehmpath
├── [+] export aws credentials
├── [+] aws cloudformation continue-update-rollback
├── [+] wait for completion
└── [+] report result
```

# nitpick.1: Decode-friction in planned .sh step descriptions

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.prefer.decomposable-architecture.md

**locations**:
- .behavior/v2026_05_12.operator/3.3.1.blueprint.product.yield.md
- src/domain.roles/operator/skills/invoke.command.sh
- src/domain.roles/deployer/skills/aws.cloudformation.rollback.sh
- src/domain.roles/operator/skills/use.rds.capacity.sh

Several codepath trees list steps that would require mental simulation if implemented directly (e.g. "if --list: display commands and exit", "check stack state = UPDATE_ROLLBACK_FAILED", "wait for db capacity (up to 180s)"). These should be extracted to named transformers (e.g. shouldListCommands({ args }), isRollbackFailed({ status })) so that the .sh (or a calling orchestrator) reads as pure narrative of what happens.

**snippet**:
```text
invoke.command.sh
├── [+] parse args (--env, --name, --list, passthrough)
├── [+] validate env (prep | prod)
├── [+] discover commands from repo config
├── [+] if --list: display commands and exit
├── [+] validate command exists
```
