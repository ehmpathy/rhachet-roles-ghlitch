# blocker.1: Failhide via top-level promise catch hiding errors

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.roles/observer/skills/query.database.ts

The entrypoint uses main().catch to catch all errors, log only the message (no stack or context), and exit(1) without rethrowing. This is a classic failhide pattern that silently hides real errors. Per the rule, try/catch (or equivalent .catch) is only allowed with an allowlist of specific errors that are carefully handled while rethrowing all others. Here all errors are caught and suppressed, violating fail-fast/fail-loud and leading to hard-to-debug defects.

**snippet**:
```ts
main().catch((error) => {
  console.error('🔮 path obscured...');
  console.error('');
  console.error(`   ${error.message}`);
  process.exit(1);
});
```

# blocker.2: Plain new Error() instead of semantic ConstraintError/MalfunctionError

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.failloud.md

**locations**:
- src/domain.roles/supporter/skills/use.vpc.tunnel.ts

Code throws new Error() for absent required config without using ConstraintError (for caller-must-fix, exit 2) or MalfunctionError (for server-must-fix, exit 1). This violates failloud requirements: errors must use proper classes with full context/metadata to enable immediate diagnosis, distinguish who must fix, and set semantic exit codes. The comments even document the intended exit codes but the implementation does not use the classes that provide them.

**snippet**:
```ts
if (!account || !region) {
    throw new Error(
      'absent required config: AWS_ACCOUNT_ID and AWS_REGION (or AWS_DEFAULT_REGION)',
    );
  }
```

# blocker.4: Constraint errors will not produce exit code 2

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.exit-code-semantics.md

**locations**:

The getTunnelConfig function throws for absent required args (constraint cases per the file's own guarantee comment), but uses plain Error. Uncaught errors in this context will result in exit 1 (malfunction) instead of exit 2 (constraint). Per the rule, constraint errors must use exit 2 so callers know to fix input rather than retry. The skill documents the intended semantics but does not implement them.

    );
