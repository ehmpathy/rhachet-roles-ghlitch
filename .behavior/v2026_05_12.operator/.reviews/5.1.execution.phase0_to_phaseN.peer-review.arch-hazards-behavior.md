# blocker.1: Hidden dependencies on external state

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.roles/observer/skills/query.database.ts
- src/domain.roles/supporter/skills/use.vpc.tunnel.ts

Procedures like main() and getTunnelConfig() (and callers getProviders/getResources) depend on process.argv, process.env, process.stdin, child_process.execSync for git, dynamic import based on git root, and fs.readFile without these being passed explicitly via input or context parameters. This creates non-determinism because the same function call can produce different results based on external environment state (e.g. current git repo, env vars set by wrappers, stdin availability, or presence of DECLASTRUCT_CONFIG). It also hides side effects and makes behavior unpredictable across different run contexts, directories, or on retries, violating the requirement for code that behaves predictably under all conditions. External state dependencies without explicit contracts are exactly the class of behavior hazards the rule targets.

**snippet**:
```ts
const main = async () => {
  // dynamic import from git root (allows use in any repo with getDatabaseConnection)
  const gitRoot = execSync('git rev-parse --show-toplevel', {
    encoding: 'utf-8',
  }).trim();
  const { getDatabaseConnection } = await import(
    path.join(gitRoot, 'src/utils/database/getDatabaseConnection')
  );

  // parse args
  const args = process.argv.slice(2);
  let sql = '';
  let format = 'table';

  for (let i = 0; i < args.length; i++) {
    const arg = args[i]!;
    if (arg === '--sql') {
      const value = args[++i];
      if (value === '@stdin') {
        // read from stdin
        const chunks: Buffer[] = [];
        for await (const chunk of process.stdin) {
          chunks.push(chunk);
        }
        sql = Buffer.concat(chunks).toString('utf-8').trim();
      } else {
        sql = value ?? '';
      }
    } else if (arg === '--format') {
      format = args[++i] ?? 'table';
    } else if (
      ['--env', '--limit', '--repo', '--role', '--skill'].includes(arg)
    ) {
      i++; // skip value
    } else if (arg === '--') {
      // skip separator
    }
  }

  // validate sql
  if (!sql) {
    console.error('🔮 path obscured...');
    console.error('');
    console.error('   absent required arg: --sql');
    process.exit(2);
  }

  // connect to database in readonly mode
  const dbConnection = await getDatabaseConnection({ mode: 'readonly' });

  try {
    const result = await dbConnection.query({ sql });
    ...
  } finally {
    await dbConnection.end();
  }
};
```

# blocker.2: Error-prone manual argument parsing

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.roles/observer/skills/query.database.ts

The arg parsing loop uses manual index incrementing (const value = args[++i]; i++ to skip) to consume values for --sql, --format and other flags. This is fragile and can produce unexpected defaults: sql set to undefined/empty string or to a subsequent flag name if a value is absent or if @stdin appears; format silently defaults to 'table'; stdin read happens at parsing time with no bounds checking or error handling for malformed input. This creates order-dependent and non-deterministic outcomes for the same logical args in different positions or with missing values, exactly the kind of unpredictable runtime behavior the rule forbids.

# blocker.3: Inconsistent config validation leading to unexpected values

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.roles/supporter/skills/use.vpc.tunnel.ts

getTunnelConfig loads TunnelConfig from DECLASTRUCT_CONFIG json file with direct JSON.parse and return (no required-field checks or sanitization), while the env-var fallback path explicitly throws on missing account/region/bastion/cluster. When the file path is present but incomplete, config.account, config.region, config.bastion etc. will be undefined and passed straight into DeclaredAwsVpcTunnel.as(...) and RefByUnique.as(...). Additionally port: port ? parseInt(port, 10) : 5432 can produce NaN with no validation or error. This produces silent unexpected defaults and non-deterministic behavior depending on whether the config file or env path is taken, violating predictable behavior under all conditions.

**snippet**:
```ts
const getTunnelConfig = async (): Promise<TunnelConfig> => {
  // check for config file path
  const configPath = process.env.DECLASTRUCT_CONFIG;
  if (configPath) {
    const fs = await import('node:fs/promises');
    const content = await fs.readFile(configPath, 'utf-8');
    return JSON.parse(content) as TunnelConfig;
  }

  // fall back to env vars (set by shell wrapper)
  const account = process.env.AWS_ACCOUNT_ID;
  const region = process.env.AWS_REGION ?? process.env.AWS_DEFAULT_REGION;
  const bastion = process.env.VPC_TUNNEL_BASTION;
  const cluster = process.env.VPC_TUNNEL_CLUSTER;
  const port = process.env.VPC_TUNNEL_PORT;
  const host = process.env.VPC_TUNNEL_HOST;

  if (!account || !region) {
    throw new Error(
      'absent required config: AWS_ACCOUNT_ID and AWS_REGION (or AWS_DEFAULT_REGION)',
    );
  }

  if (!bastion || !cluster) {
    throw new Error(
      'absent required config: --bastion and --cluster (or --config path)',
    );
  }

  return {
    account,
    region,
    bastion: { exid: bastion },
    cluster: { name: cluster },
    local: {
      port: port ? parseInt(port, 10) : 5432,
      host: host || undefined,
    },
  };
};
```
