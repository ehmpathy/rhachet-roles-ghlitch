## summary

promote 14 generic skills from ahbode/svc-jobs into rhachet-roles-ghlitch, organized into three roles:

| role | skills | purpose |
|------|--------|---------|
| observer | query.database, query.cloudwatch.logs, query.cloudwatch.metrics, aws.ssm.param.check | readonly investigation |
| operator | invoke.command, invoke.vital, use.rds.capacity, use.vpc.tunnel, use.testdb | operational support |
| deployer | deploy, aws.cloudformation.status, aws.cloudformation.rollback, provision.database, provision.terraform | infrastructure changes |

key changes from source:
- rename skills: `show.*` → `query.*`, `cloudformation.*` → `aws.cloudformation.*`, `terraform` → `provision.terraform`
- standardize env names: `test, prep, prod` (not `dev`)

### glossary

| term | definition |
|------|------------|
| vital | a health check endpoint that returns system liveness — pass/fail with diagnostic info. analogous to kubernetes liveness probes or AWS health checks. vitals are registered in repo config and invoked via `invoke.vital`. |
| command | a predefined operation registered in repo config that can be invoked remotely. commands differ from vitals: they perform actions (mutations, jobs, maintenance) rather than return status. |

### skill name semantics

the `query.*` prefix is semantically consistent — it means "request/retrieve data" in all contexts:
- `query.database` — retrieve data from database via SQL
- `query.cloudwatch.logs` — retrieve data from CloudWatch Logs via filter
- `query.cloudwatch.metrics` — retrieve data from CloudWatch Metrics via API

this is analogous to HTTP GET or database SELECT — the verb "query" consistently means "read without side effects". the target resource varies (database, logs, metrics), but the operation semantics are identical: retrieve data.

contrast with overloaded terms (avoided):
- ~~`fetch`~~ — ambiguous (could mean network fetch, data retrieval, or eager load)
- ~~`get`~~ — reserved for domain.operations pattern (get-set-gen)
- ~~`show`~~ — imperative verb that implies display, not retrieval

### implementation source

**port implementations directly from ahbode/svc-jobs** — these are battle-tested, production-proven skills. do not rewrite from scratch.

steps:
1. copy the implementation from svc-jobs
2. rename as specified (e.g., `show.logs.sh` → `query.cloudwatch.logs.sh`)
3. update keyrack to `--owner ehmpath`
4. update env names to `test | prep | prod`
5. add role-specific mascot header (see below)

### skill header requirements

each skill must have a consistent mascot+artifact header for its role:

| role | mascot | header vibe |
|------|--------|-------------|
| observer | 🔮 | crystal ball sees all, watches patterns |
| operator | 🦺 | safety vest supports operations |
| deployer | ⛵ | ship sails to production |

header format (bash):
```bash
#!/usr/bin/env bash
######################################################################
# 🔮 query.cloudwatch.logs — observe log streams
#
# .what = retrieve cloudwatch logs for a lambda function
#
# .why  = enables log investigation without aws console
#
# usage:
#   rhx query.cloudwatch.logs --env prep --lambda myFunc --since 1h
#
# guarantee:
#   - exit 0 = success
#   - exit 1 = malfunction (aws error)
#   - exit 2 = constraint (bad input)
######################################################################
```

header format (typescript):
```typescript
/**
 * 🦺 use.vpc.tunnel — establish secure database tunnel
 *
 * .what = creates ssm tunnel to rds endpoint via ec2 bastion
 *
 * .why  = enables local database access without public rds exposure
 */
```

### test strategy: AWS resources

these skills depend on AWS services that may not exist in the ehmpathy demo account. we use a tiered strategy:

#### tier 1: hermetic tests (no AWS required)

| what | how | coverage |
|------|-----|----------|
| arg parse | unit tests | --env, --sql, --format validation |
| env validation | unit tests | test/prep/prod checks |
| output format | unit tests | table/csv/json transformers |
| error messages | unit tests | ConstraintError, MalfunctionError |

#### tier 2: PATH injection tests (fake `aws` CLI)

per rhachet-roles-ehmpathy pattern (`howto.mock-cli-via-path`):

| what | how | coverage |
|------|-----|----------|
| aws CLI calls | fake `aws` binary prepended to PATH | cloudwatch, ssm, cloudformation |
| keyrack unlock | fake `rhx` binary | credential flow |
| output parse | canned responses | JSON → formatted output |

**guarantee**: fake binaries return realistic canned responses. responses are captured from real `aws` CLI runs and stored as fixtures.

#### tier 3: real AWS tests (documented exception)

per `rule.forbid.integration.mocks`, real integration tests require real resources. when resources are absent:

```typescript
/**
 * .mock = none (real AWS required)
 * .why = RDS connection, SSM tunnel, CloudWatch queries need real infra
 * .real = manual verification documented below
 * .quarterly = run full suite against ahbode/svc-jobs test account
 */
if (!process.env.AWS_ACCOUNT_ID) {
  throw new ConstraintError('AWS_ACCOUNT_ID required', {
    hint: 'run against ahbode/svc-jobs test account or provision test resources',
    doc: 'docs/test-aws-skills.md',
  });
}
```

#### accuracy guarantee

1. **fixtures from production** — canned responses captured from real svc-jobs AWS account
2. **fixture refresh schedule** — quarterly refresh to catch AWS API changes
3. **quarterly integration runs** — full suite against real AWS (ahbode/svc-jobs test account)
4. **snapshot drift detection** — CI fails if fixture format diverges from expected

#### test infrastructure

| resource | purpose | provisioned where |
|----------|---------|-------------------|
| RDS instance | query.database | ahbode/svc-jobs test account |
| CloudWatch log group | query.cloudwatch.logs | ahbode/svc-jobs test account |
| SSM-enabled EC2 | use.vpc.tunnel | ahbode/svc-jobs test account |
| CloudFormation stack | aws.cloudformation.* | ahbode/svc-jobs test account |

**note**: `use.testdb` is hermetic — it uses local docker, no AWS needed.
