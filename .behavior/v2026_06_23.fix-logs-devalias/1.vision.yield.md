# vision: fix-logs-devalias

## the outcome world

### before

```bash
# user queries prep logs
rhx aws.cloudwatch.query --env prep --lambda createJob

# 🐈 belay that...
# 🔮 aws.cloudwatch.query
#    ├─ log group not found: /aws/lambda/svc-foo-prep-createJob
#    │
#    └─ available log groups:
#       /aws/lambda/svc-foo-dev-createJob   <-- the actual log group!
```

user stares. they know the lambda was deployed. they just can't find the logs because the service was deployed with `-dev` (the historic convention) instead of `-prep` (the current convention).

### after

```bash
# user queries prep logs
rhx aws.cloudwatch.query --env prep --lambda createJob

# 🐈 caught it!
# 🔮 aws.cloudwatch.query
#    └─ observed
#
# 2026-06-23 14:32:01 | {"level":"INFO","message":"createJob invoked"}
```

it just works. the skill searches both `-prep` and `-dev` log groups, finding whichever is present.

### the "aha" moment

no more mental gymnastics. no more remembering which services use `-dev` vs `-prep`. the tool handles the historic alias automatically.

## user experience

### usecases

| usecase | user goal |
|---------|-----------|
| debug failing request | find logs for a lambda in prep |
| investigate production incident | find logs across all lambdas in prod |
| check metrics | see invocation counts for prep environment |

### contract

**input unchanged:**
```bash
rhx aws.cloudwatch.query --env prep --lambda createJob
rhx aws.cloudwatch.query --env prep --list
rhx query.cloudwatch.metrics --env prep
```

**output enhanced:**
- when `--env prep` is passed, search both `-prep` and `-dev` suffixes
- when `--list` is used, show log groups from both conventions
- when querying metrics, aggregate from both naming conventions

### timeline

1. user runs `rhx aws.cloudwatch.query --env prep --lambda foo`
2. skill searches for `/aws/lambda/$PREFIX-prep-foo`
3. if not found, skill searches for `/aws/lambda/$PREFIX-dev-foo`
4. if found via either path, query proceeds
5. output shows which log group was actually queried

## mental model

### how users would describe this to a friend

> "when i query prep logs, it automatically checks both -prep and -dev log groups. some of our older services still use -dev as the environment name, so this saves me from having to remember which is which."

### analogies

like how `localhost` and `127.0.0.1` are aliases for the same thing — when you type one, the system understands you mean "the local machine", regardless of which syntax you used.

### terms

| user says | system interprets |
|-----------|------------------|
| `--env prep` | search `-prep` first, fallback to `-dev` |
| `--env test` | search `-test` only (no alias) |
| `--env prod` | search `-prod` only (no alias) |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| find logs without knowing naming convention | ✓ yes |
| no breaking changes to extant commands | ✓ yes |
| transparent to user which convention was used | ✓ yes (shown in output) |

### pros

- **zero learning curve** — extant commands work unchanged
- **backwards compatible** — services using `-prep` still work
- **transparent** — user sees which log group was actually queried
- **simple mental model** — "prep means prep, even if the old name was dev"

### cons

- **minor perf hit** — potentially two AWS calls instead of one when fallback is needed
- **implicit behavior** — user might not realize they're querying a `-dev` log group
- **only affects prep** — doesn't solve if other aliases exist (none known)

### edgecases

| edgecase | handling |
|----------|----------|
| both `-prep` and `-dev` exist for same lambda | prefer `-prep` (the canonical name) |
| neither is present | fail with current error message |
| `--list` mode | show all log groups from both conventions |

## what is awkward?

### implicit behavior

the fallback from `-prep` to `-dev` is invisible magic. users might not realize they're querying a log group with a different name than they specified. this could cause confusion when:
- debugging deployment issues ("why does my --env prep hit a -dev log group?")
- discussing with teammates ("check the prep logs" vs "check the dev logs")

**mitigation:** emit the actual log group name in output, possibly with a hint when fallback was used.

### asymmetric aliases

only `prep` has an alias (`dev`). `test` and `prod` don't. this asymmetry might confuse users who expect consistent behavior:
- "if prep falls back to dev, does prod fall back to something?"

**mitigation:** document clearly that `-dev` is a historic artifact, not a general pattern.

### multi-skill scope

two skills are affected:
1. `aws.cloudwatch.query.sh`
2. `query.cloudwatch.metrics.sh`

each must implement the same fallback logic. if we add more observer skills later, they need the same pattern.

**mitigation:** consider extracting a shared function `get_env_suffixes()` that returns `["prep", "dev"]` for prep and `["$ENV"]` for others.
