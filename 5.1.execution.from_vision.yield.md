## blockers — both RESOLVED (both were prior, NOT caused by this work)

### 1. keyrack misconfig blocks the integration harness — RESOLVED

- `rhx git.repo.test --what integration` unconditionally runs `rhx keyrack unlock --owner ehmpath --env test` before any test (git.repo.test.sh:1031, fatal on failure).
- it fails: `ConstraintError: key lacks awsSsoUsername metadata; re-set the key` for `AWS_PROFILE`.
- my tests need **zero** credentials (they run shell skills against temp dirs), but the harness blocks before it reaches them.
- fix is human-only: `rhx keyrack set --owner ehmpath --env test --key AWS_PROFILE --vault aws.config` (needs the AWS SSO username).

- resolution: human fixed the key (`rhx keyrack set ... AWS_PROFILE`) + completed the SSO unlock; the harness then unlocked cleanly and ran my tests → **16/16 passed**.

### 2. @aws-crypto cycles fail lint:cycles — RESOLVED

- `test:lint:cycles` (dpdm) reported circular deps in `@aws-crypto/crc32` + `@aws-crypto/crc32c`.
- these are third-party node_modules cycles, pulled in by the **pre-staged** dep bump (`rhachet` 1.41.22→1.42.4 etc., staged before this session).
- my diff adds only `.sh` files + one `.test.ts` (excluded from dpdm's `src/**/!(*.test).ts` scan), so it cannot introduce these.
- root cause of the *local* failure was also an absent host `yq` binary — without it the lint command silently dropped the whole exclude list (every prior exclude leaked too).
- resolution: human installed `yq`, and i added `aws-crypto` to `.dpdmrc.yaml` exclude (alongside the prior `aws-sdk`). lint now **passes**.

1. **terraform `init` ungated** — the read-only allowlist treats `terraform init` as a pass-through (it's a prerequisite for `plan`). but `init` *can* migrate backend state on a backend-config change. left ungated so plan stays open; flag if you'd rather gate it (one-word delete from the allowlist in `provision.terraform.sh:152`).
2. **output style divergence** — the new `*.uses` skills source a shared `uses.output.sh`; the peer ghlitch skills (`deploy.sh`, `provision.*.sh`) inline their `echo "🐈..."` lines. mine matches the reference engines and suits a 7-file engine, but it's a mild inconsistency. a future cleanup could lift one shared ghlitch `output.sh` for all deployer skills — out of scope for this wish.
3. **org precedence + "org allow never grants"** (clarified by the wisher; implemented & tested) — org policy is **block-only in effect**:
   - **org `block`** = a HARD FREEZE. wins over a local `allow` (a local grant cannot bypass it). this honors the vision's threat model: a freeze "controlled by someone other than the actor".
   - **org `allow`** = does **NOT** grant prod on its own. it ONLY lifts an org freeze (e.g. overrides an `@all` block for one org). a repo still needs its **own local grant** to reach prod.
   - **local grant** = the ONLY path that permits prod (when not frozen).
   - precedence: `global-block → org-block → local(allow/block) → unset`. there is no longer an `allowed:org` outcome.
   - made unmistakable in the engine header (`uses.operations.sh` `decide_uses`), the org skill header + output (`allow` prints "does NOT grant prod — a local grant is still required"), and the gate (`uses.check.sh`).
   - covered by tests: case7 (org allow alone → blocked), case7b (org block beats local allow → blocked), case7c (org allow + local allow → allowed), case7d (org allow lifts an @all freeze but still needs a local grant: without local → blocked, with local → allowed).
