# Blueprint: setLambdaObservability

## Decision: NOT RECOMMENDED

**After analysis, we decided NOT to implement Lambda Insights.**

### Why Lambda Insights is Not Needed

We already have `getLambdaExpenseEvaluator.sh` which provides the same cost optimization insights **without any additional AWS costs**:

**What `getLambdaExpenseEvaluator.sh` already provides:**
- ✅ Memory utilization analysis (via CloudWatch Logs `REPORT` lines - FREE)
- ✅ Duration metrics (via CloudWatch Metrics - FREE within limits)
- ✅ Invocation counts (via CloudWatch Metrics - FREE within limits)
- ✅ Cost calculations per function
- ✅ Architecture detection (x86 vs ARM)
- ✅ Identifies over/under-provisioned functions
- ✅ **Zero additional cost** - uses existing free-tier CloudWatch data

**What Lambda Insights would add:**
- ❌ Same memory utilization data (redundant)
- ❌ Same duration patterns (redundant)
- ❌ CPU utilization (marginal value for cost optimization)
- ❌ Network I/O (marginal value for cost optimization)
- ❌ **Costs $25-150/month** for ongoing monitoring

### Cost-Benefit Analysis

**Lambda Insights Cost:**
- 50 functions @ avg 100K invocations/month: **~$25-50/month**
- 50 functions @ avg 1M invocations/month: **~$100-150/month**

**Value Add Over Existing Solution:**
- Minimal - the key optimization signals (memory utilization, duration, invocations) are already captured
- The existing solution queries CloudWatch Logs for actual max memory usage at zero cost
- CPU and network metrics have limited cost optimization value for Lambda

**Conclusion:**
Lambda Insights is **not worth the cost** when `getLambdaExpenseEvaluator.sh` already provides the critical data for free.

## Implementation Blueprint

### 1. Script Structure

**Location**: `src/roles/budgeter/skills/aws.lambda/setLambdaObservability.sh`

**Signature**:
```bash
./setLambdaObservability.sh --mode=<prep|exec> [--function <name>] [--output <dir>] [--region <region>]
```

### 2. Core Components

#### A. Argument Parsing
- `--mode=<prep|exec>`: Required mode
  - `prep`: Analyze and report what would change (no modifications)
  - `exec`: Execute changes and apply observability settings
- `--function <name>`: Optional, target specific function(s) (repeatable flag)
- `--output <dir>`: Output directory (default: `.rhachet/setLambdaObservability/<account>/<timestamp>`)
- `--region <region>`: AWS region (default: current AWS CLI config region)

#### B. Discovery Phase
```bash
# list all lambda functions (or specified ones)
aws lambda list-functions --output json

# for each function, check current configuration
aws lambda get-function-configuration \
  --function-name <name> \
  --query 'Layers[]' \
  --output json
```

Check for Lambda Insights layer:
- Layer ARN pattern: `arn:aws:lambda:<region>:580247275435:layer:LambdaInsightsExtension`
- Environment variable: `LAMBDA_INSIGHTS_LOG_LEVEL`

#### C. Configuration Phase
For functions lacking Lambda Insights:

```bash
# get current function config
CURRENT_LAYERS=$(aws lambda get-function-configuration \
  --function-name <name> \
  --query 'Layers[].Arn' \
  --output json)

# determine insights layer ARN for region
INSIGHTS_LAYER="arn:aws:lambda:${REGION}:580247275435:layer:LambdaInsightsExtension:21"

# add insights layer (preserve existing layers)
aws lambda update-function-configuration \
  --function-name <name> \
  --layers "${CURRENT_LAYERS[@]}" "$INSIGHTS_LAYER" \
  --environment "Variables={LAMBDA_INSIGHTS_LOG_LEVEL=info}"
```

Also enable enhanced monitoring:
```bash
# ensure function has proper IAM permissions
# check if execution role has CloudWatchLambdaInsightsExecutionRolePolicy

# attach managed policy if missing
aws iam attach-role-policy \
  --role-name <execution-role-name> \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchLambdaInsightsExecutionRolePolicy
```

### 4. Cost Analysis

#### Lambda Insights Pricing (as of 2025)

**Per Function Costs:**
- **1M invocations/month**: ~$2.92/month
- **100K invocations/month**: ~$0.50-1.00/month
- **10K invocations/month**: ~$0.10-0.30/month

**Cost Components:**
1. **Lambda Extension Overhead**: 1-5ms added per invocation (charged at Lambda compute rates)
2. **CloudWatch Custom Metrics**: $0.30/metric/month (first 10K), $0.10 after; ~8 metrics per function
3. **CloudWatch Logs**: $0.50/GB (first 10 TB), $0.25/GB beyond
4. **Logs Insights Queries**: $0.005/GB scanned (only when querying)

**Estimated Total Cost:**
- 50 functions @ avg 100K invocations/month: **~$25-50/month**
- 50 functions @ avg 1M invocations/month: **~$100-150/month**

#### Cost vs Savings Trade-off

**Break-even Analysis:**
- Reducing one function from 1024MB → 512MB saves ~$8/month (@ 1M invocations)
- Migrating one function from x86 → ARM saves ~20% compute cost
- **ROI positive if you optimize 3-6 functions**

**⚠️ Recommendation: Temporary Deployment Only**

Due to ongoing costs, Lambda Insights should be:
- ✅ Enabled temporarily (7-30 days) for data collection
- ✅ Used to identify optimization opportunities
- ✅ Disabled after gathering sufficient metrics
- ✅ Re-enabled periodically (quarterly) to validate optimizations
- ❌ Not left permanently enabled across all functions

### 5. Observability Benefits

Lambda Insights provides metrics for:
- **Memory utilization**: Actual vs allocated memory
- **Duration patterns**: P50, P90, P99 latencies
- **Cold start frequency**: Identify functions with frequent cold starts
- **Concurrent execution**: Peak concurrency vs configured limits
- **CPU utilization**: Detect CPU-bound vs I/O-bound functions
- **Network I/O**: Bytes sent/received

These metrics enable cost optimization by:
- Right-sizing memory allocation (under/over-provisioned functions)
- Identifying candidates for provisioned concurrency
- Detecting candidates for ARM/Graviton migration
- Finding idle functions that can be deprecated

### 6. Implementation Guidelines

Follow repository coding standards:
- **fail-fast**: Early exits with clear error messages
- **idempotency**: Check before apply, safe to re-run
- **immutability**: Never mutate AWS resources in `--mode=prep`

- **compressed style**: Clear comments explaining "why" not "what"
- **mode-driven**: Require explicit `--mode` flag, no default mode

### 7. Integration Points

The script should integrate with existing budgeter skills:
- Output compatible with `getLambdaExpenseEvaluator.sh` for before/after analysis
- Reference in `getLambdaExpenseEvaluator.sh` output as "Next Steps"
- Query Lambda Insights metrics via CloudWatch Logs Insights

### 8. CloudWatch Logs Insights Queries

After enabling Lambda Insights, functions generate enhanced metrics in CloudWatch Logs.

Example query for memory optimization:
```
filter @type = "platform.runtimeDone"
| fields @timestamp, @memorySize / 1000000 as provisioned_memory_mb,
         @maxMemoryUsed / 1000000 as used_memory_mb,
         (@maxMemoryUsed / @memorySize) * 100 as memory_utilization
| stats avg(memory_utilization) as avg_util,
        max(used_memory_mb) as max_used_mb,
        min(provisioned_memory_mb) as provisioned_mb
        by @functionName
| filter avg_util < 60
```

### 9. Testing Strategy

**Prep mode (`--mode=prep`) validation**:
- Enumerate functions correctly
- Detect missing insights accurately
- Generate correct diff reports
- No AWS modifications

**Exec mode (`--mode=exec`) validation**:
- Layers added successfully
- IAM policies attached correctly
- Verification confirms changes
- Idempotent re-runs

## Summary

**Decision: Do NOT implement this blueprint.**

After creating this blueprint, we realized that `getLambdaExpenseEvaluator.sh` already provides all the necessary cost optimization insights without any additional AWS charges. Lambda Insights would add $25-150/month in costs for minimal additional value.

**Recommendation:** Continue using `getLambdaExpenseEvaluator.sh` for cost-free Lambda optimization analysis. Only consider Lambda Insights if you need detailed CPU profiling or network I/O metrics for performance debugging (not cost optimization).
