# Blueprint: S3 Bucket Observability Provisions

## Outcome
Establish minimum-cost, near-real-time observability infrastructure for S3 buckets to:
- Generate tree structure views with directory sizes
- Track object access patterns (last-used: 0d, 1d, 3d, 30d, 90d, 180d+)
- Identify storage tier optimization opportunities

**Technology**: S3 Metadata (Apache Iceberg tables, announced re:Invent 2024, enhanced July 2025)
**Cost target**: ~$0.45 per 100M objects/month (S3 Metadata + Athena queries)
**Data freshness**: 1 hour (live inventory) + near real-time (journal table)
**Implementation time**: 2 hours (vs 48+ hours with S3 Inventory)

## Why S3 Metadata Over S3 Inventory?

S3 Metadata is the modern replacement for S3 Inventory:

| Feature | S3 Metadata (This Blueprint) | S3 Inventory (Legacy) |
|---------|------------------------------|----------------------|
| **First data available** | 1 hour | 24-48 hours |
| **Update frequency** | Hourly + real-time changes | Daily/Weekly |
| **Format** | Apache Iceberg (columnar) | CSV/Parquet |
| **Query performance** | Optimized (predicate pushdown) | Full table scans |
| **Change tracking** | Journal table (all changes) | Snapshot only |
| **Cost (100M objects)** | ~$0.25/month | ~$0.25/month |
| **Setup complexity** | 3 steps | 5 steps |

**Recommendation**: Use S3 Metadata exclusively. S3 Inventory is legacy.

## Prerequisites
- AWS CLI v2.18.17 or later (V2 API required)
- jq (JSON processor for parsing AWS CLI outputs)
- Analysis bucket will be auto-created: `ghlitch-${ACCOUNT_ID_HASH}-objects` where `ACCOUNT_ID_HASH` is first 7 characters of SHA256 hash of account ID
- **S3 Tables bucket**: Customer-managed bucket with AWS analytics integration (auto-created: `ghlitch-${ACCOUNT_ID_HASH}-s3metadata`)
- **S3 permissions**: `s3:CreateBucket`, `s3:CreateBucketMetadataConfiguration`, `s3:GetBucketMetadataConfiguration`, `s3:GetObject`, `s3:PutObject`
- **S3 Tables permissions**: `s3tables:CreateTableBucket`, `s3tables:CreateTable`, `s3tables:GetTable`, `s3tables:GetTableData`, `s3tables:PutTableBucketPolicy`
- **Athena permissions**: `athena:StartQueryExecution`, `athena:GetQueryExecution`, `athena:GetQueryResults`
- **Glue permissions**: `glue:CreateDatabase`, `glue:GetDatabase`, `glue:CreateTable`, `glue:GetTable`
- **Lake Formation permissions**: `lakeformation:GrantPermissions`, `lakeformation:GetDataAccess`

## Architecture

```
┌─────────────────────┐
│   Source Bucket     │
└──────────┬──────────┘
           │
           ├─→ [S3 Metadata V2 API] ──→ Apache Iceberg Tables
           │                            │
           │                            ├─→ Journal Table (near real-time changes)
           │                            └─→ Inventory Table (hourly refresh)
           │
           └─→ Stored in: AWS-Managed S3 Tables Bucket
                        TableBucketArn: arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/aws-s3
                        Namespace: b_${SOURCE_BUCKET} (auto-assigned)
                        Journal Table: journal (ACTIVE immediately)
                        Inventory Table: inventory (BACKFILLING, ready in ~1 hour)

              Query via Athena for:
              • Tree structure + sizes
              • Storage class distribution
              • Access patterns (via last_modified_date)
              • Optimization opportunities

Optional (for validation):
  [Storage Class Analysis] → CSV reports (30 day warmup)
  [Storage Lens] → Dashboard (free tier, org-wide)
  [CloudWatch] → Basic metrics (free, automatic)
```

## Implementation Steps

### Step 1: Enable S3 Metadata on Source Bucket (V2 API)
**Timeline**: 1 hour for initial data, ongoing updates within minutes
**Cost**: ~$0.0025 per million objects/month (similar to S3 Inventory)

Enable S3 Metadata to create Apache Iceberg tables with near real-time updates. AWS V2 API automatically creates and manages the S3 Tables bucket:

```bash
#!/bin/bash
SOURCE_BUCKET="your-bucket-name"

# Get AWS account and region
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=$(aws configure get region)

# Enable S3 Metadata configuration using V2 API
# AWS automatically creates S3 Tables bucket and manages tables
aws s3api create-bucket-metadata-configuration \
  --bucket "${SOURCE_BUCKET}" \
  --metadata-configuration '{
    "JournalTableConfiguration": {
      "RecordExpiration": {
        "Expiration": "DISABLED"
      },
      "EncryptionConfiguration": {
        "SseAlgorithm": "AES256"
      }
    },
    "InventoryTableConfiguration": {
      "ConfigurationState": "ENABLED",
      "EncryptionConfiguration": {
        "SseAlgorithm": "AES256"
      }
    }
  }' \
  --region "${REGION}"

echo "S3 Metadata enabled for ${SOURCE_BUCKET}"
```

**Validation**:
```bash
# Verify metadata configuration and get table details
aws s3api get-bucket-metadata-configuration \
  --bucket "${SOURCE_BUCKET}" \
  --region "${REGION}"

# Example output shows:
# - TableBucketArn: arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/aws-s3
# - Namespace: b_${SOURCE_BUCKET}
# - Journal Table: journal (ACTIVE)
# - Inventory Table: inventory (BACKFILLING)
```

**What AWS Creates Automatically**:
- **AWS-Managed S3 Tables Bucket**: `arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/aws-s3`
- **Namespace**: `b_${SOURCE_BUCKET}` (prefixed with `b_`)
- **Journal Table**: `journal` (status: ACTIVE immediately)
- **Inventory Table**: `inventory` (status: BACKFILLING for ~1 hour, then ACTIVE)
- **Apache Iceberg format**: Industry-standard table format, queryable via Athena, Spark, etc.
- **Metadata includes**: key, size, last_modified_date, storage_class, intelligent_tiering_access_tier, etag, version_id

### Step 2: Register S3 Tables in AWS Glue Data Catalog
**Timeline**: 5 minutes
**Cost**: Free (Glue Data Catalog storage is free for first million objects)

While S3 Metadata V2 creates Iceberg tables automatically, you must register them in Glue to query via Athena:

```bash
#!/bin/bash
SOURCE_BUCKET="your-bucket-name"

# Get AWS account and region
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
ACCOUNT_ID_HASH=$(echo -n "${ACCOUNT_ID}" | sha256sum | cut -c1-7)
REGION=$(aws configure get region)
BUCKET_ANALYSIS="ghlitch-${ACCOUNT_ID_HASH}-objects"

# Ensure analysis bucket exists for Athena results
aws s3 mb "s3://${BUCKET_ANALYSIS}" --region "${REGION}" 2>/dev/null || echo "Analysis bucket exists"

# Get inventory table ARN from S3 Metadata configuration
CONFIG=$(aws s3api get-bucket-metadata-configuration --bucket "${SOURCE_BUCKET}" --region "${REGION}")
INVENTORY_ARN=$(echo "$CONFIG" | jq -r '.GetBucketMetadataConfigurationResult.MetadataConfigurationResult.InventoryTableConfigurationResult.TableArn')
TABLE_BUCKET_ARN=$(echo "$CONFIG" | jq -r '.GetBucketMetadataConfigurationResult.MetadataConfigurationResult.DestinationResult.TableBucketArn')
NAMESPACE=$(echo "$CONFIG" | jq -r '.GetBucketMetadataConfigurationResult.MetadataConfigurationResult.DestinationResult.TableNamespace')

echo "Inventory Table ARN: ${INVENTORY_ARN}"

# Create Glue database if it doesn't exist
aws glue create-database \
  --database-input "{
    \"Name\": \"s3_metadata\",
    \"Description\": \"S3 Metadata tables for cost optimization\"
  }" \
  --region "${REGION}" 2>/dev/null || echo "Database s3_metadata already exists"

# Get table location from S3 Tables
# Note: S3 Tables store data in their own managed location
# We need to create an Iceberg table in Glue pointing to this location
TABLE_LOCATION="s3tables://${INVENTORY_ARN}"
GLUE_TABLE_NAME=$(echo "${SOURCE_BUCKET}" | tr '-' '_' | tr '.' '_')_inventory

# Create Glue Iceberg table
aws glue create-table \
  --database-name "s3_metadata" \
  --table-input "{
    \"Name\": \"${GLUE_TABLE_NAME}\",
    \"Description\": \"S3 Metadata inventory table for ${SOURCE_BUCKET}\",
    \"TableType\": \"EXTERNAL_TABLE\",
    \"Parameters\": {
      \"table_type\": \"ICEBERG\",
      \"metadata_location\": \"${TABLE_LOCATION}\"
    },
    \"StorageDescriptor\": {
      \"Location\": \"${TABLE_LOCATION}\",
      \"InputFormat\": \"org.apache.iceberg.mr.hive.HiveIcebergInputFormat\",
      \"OutputFormat\": \"org.apache.iceberg.mr.hive.HiveIcebergOutputFormat\",
      \"SerdeInfo\": {
        \"SerializationLibrary\": \"org.apache.iceberg.mr.hive.HiveIcebergSerDe\"
      }
    }
  }" \
  --region "${REGION}"

echo "Glue table created: s3_metadata.${GLUE_TABLE_NAME}"
echo "Query in Athena: SELECT * FROM s3_metadata.${GLUE_TABLE_NAME} LIMIT 10;"
```

**Note**: The `s3tables://` protocol requires Glue catalog registration. Direct querying without Glue is not yet supported.

## Analysis Workflows

### Setup: Query S3 Metadata Tables Directly

S3 Metadata creates Apache Iceberg tables automatically. Athena can query them directly using the `s3tables://` protocol - **no Glue registration required**.

**Setup helper script**:
```bash
#!/bin/bash
SOURCE_BUCKET="your-bucket-name"

# Get AWS account and region
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
ACCOUNT_ID_HASH=$(echo -n "${ACCOUNT_ID}" | sha256sum | cut -c1-7)
REGION=$(aws configure get region)
BUCKET_ANALYSIS="ghlitch-${ACCOUNT_ID_HASH}-objects"
ATHENA_OUTPUT="s3://${BUCKET_ANALYSIS}/budget/stats/athena-results/"

# Get table ARNs from the metadata configuration
CONFIG=$(aws s3api get-bucket-metadata-configuration --bucket "${SOURCE_BUCKET}" --region "${REGION}")
JOURNAL_ARN=$(echo "$CONFIG" | jq -r '.GetBucketMetadataConfigurationResult.MetadataConfigurationResult.JournalTableConfigurationResult.TableArn')
INVENTORY_ARN=$(echo "$CONFIG" | jq -r '.GetBucketMetadataConfigurationResult.MetadataConfigurationResult.InventoryTableConfigurationResult.TableArn')

echo "=== S3 Metadata Query Setup (V2 API) ==="
echo "Source Bucket: ${SOURCE_BUCKET}"
echo "Journal Table ARN: ${JOURNAL_ARN}"
echo "Inventory Table ARN: ${INVENTORY_ARN}"
echo ""
echo "Query inventory in Athena using:"
echo "  SELECT * FROM \"s3tables://${INVENTORY_ARN}\" LIMIT 10;"
echo ""
echo "Athena Output: ${ATHENA_OUTPUT}"
```

**Example query**:
```sql
-- Query S3 Metadata inventory table directly (V2 API uses AWS-managed bucket)
SELECT *
FROM "s3tables://arn:aws:s3tables:us-east-1:123456789012:bucket/aws-s3/table/c8fb2aa8-679a-4839-acc5-6d2342a07f8b"
LIMIT 10;
```

**Note**: V2 API uses AWS-managed S3 Tables bucket (`aws-s3`) with auto-generated table UUIDs. Use `get-bucket-metadata-configuration` to get the exact table ARNs.

**Why direct queries?**
- ✅ No Glue Data Catalog needed
- ✅ No table registration required
- ✅ Always queries latest metadata (no sync issues)
- ✅ Simpler setup and maintenance
- ✅ One less service to manage

**Cost**: Athena charges $5 per TB scanned. Iceberg metadata is highly optimized with columnar storage and partition pruning, typical cost: $0.01-0.05 per query.

### Workflow 1: Generate Tree Structure with Sizes

**Input**: S3 Metadata via Athena (direct query)
**Output**: Directory tree with aggregated sizes

**Note**: Replace the ARN with your actual S3 Tables ARN from setup script above

```sql
-- Query 1: Get top-level directory sizes from S3 Metadata
SELECT
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key))
    ELSE key
  END as top_level_prefix,
  COUNT(*) as object_count,
  SUM(size) as total_size_bytes,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb,
  MAX(last_modified_date) as most_recent_modification
FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
GROUP BY
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key))
    ELSE key
  END
ORDER BY total_size_bytes DESC
LIMIT 100;
```

```sql
-- Query 2: Multi-level directory tree (up to 3 levels deep)
WITH parsed_paths AS (
  SELECT
    key,
    size,
    SPLIT(key, '/') as path_parts,
    CARDINALITY(SPLIT(key, '/')) as depth
  FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
)
SELECT
  CASE
    WHEN depth >= 1 THEN path_parts[1]
    ELSE ''
  END as level_1,
  CASE
    WHEN depth >= 2 THEN path_parts[2]
    ELSE ''
  END as level_2,
  CASE
    WHEN depth >= 3 THEN path_parts[3]
    ELSE ''
  END as level_3,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as size_gb
FROM parsed_paths
GROUP BY
  CUBE(
    path_parts[1],
    path_parts[2],
    path_parts[3]
  )
HAVING level_1 != '' OR level_2 != '' OR level_3 != ''
ORDER BY size_gb DESC
LIMIT 500;
```

**Execute via CLI**:
```bash
#!/bin/bash
SOURCE_BUCKET="your-bucket-name"

# Calculate S3 Tables ARN
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
ACCOUNT_ID_HASH=$(echo -n "${ACCOUNT_ID}" | sha256sum | cut -c1-7)
REGION=$(aws configure get region)

TABLE_ARN="s3tables://arn:aws:s3tables:${REGION}:${ACCOUNT_ID}:bucket/${BUCKET_S3METADATA}/default/${SOURCE_BUCKET}-metadata"
ATHENA_OUTPUT="s3://ghlitch-${ACCOUNT_ID_HASH}-objects/budget/stats/athena-results/"

# Run Athena query directly on S3 Tables
QUERY="SELECT
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key))
    ELSE key
  END as top_level_prefix,
  COUNT(*) as object_count,
  SUM(size) as total_size_bytes,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb
FROM \"${TABLE_ARN}\"
GROUP BY 1
ORDER BY total_size_bytes DESC
LIMIT 100"

EXECUTION_ID=$(aws athena start-query-execution \
  --query-string "${QUERY}" \
  --result-configuration "OutputLocation=${ATHENA_OUTPUT}" \
  --query 'QueryExecutionId' \
  --output text)

# Wait for query to complete
aws athena wait query-succeeded --query-execution-id "${EXECUTION_ID}"

# Get results
aws athena get-query-results \
  --query-execution-id "${EXECUTION_ID}" \
  --output json > tree_structure.json

# Format as CSV
aws athena get-query-results \
  --query-execution-id "${EXECUTION_ID}" \
  --output text > tree_structure.csv
```

### Workflow 2: Detect Access Patterns

**Method A: Intelligent-Tiering Distribution** (if using IT)

Query S3 Metadata for Intelligent-Tiering tier distribution:

```sql
-- Distribution of objects by Intelligent-Tiering access tier
SELECT
  intelligent_tiering_access_tier,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb,
  ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 2) as percent_of_objects,
  CASE intelligent_tiering_access_tier
    WHEN 'FREQUENT' THEN 'Accessed within 30 days'
    WHEN 'INFREQUENT' THEN 'Not accessed 30-90 days'
    WHEN 'ARCHIVE_INSTANT' THEN 'Not accessed 90-180 days'
    WHEN 'ARCHIVE' THEN 'Not accessed 180+ days'
    WHEN 'DEEP_ARCHIVE' THEN 'Not accessed 180+ days (deep tier)'
    ELSE 'Not in Intelligent-Tiering'
  END as access_pattern
FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
WHERE storage_class = 'INTELLIGENT_TIERING'
GROUP BY intelligent_tiering_access_tier
ORDER BY total_size_gb DESC;
```

**Method B: Last Modified Date Analysis** (primary approach)

Approximate access patterns using last modified date from S3 Metadata:

```sql
-- Categorize objects by age based on last_modified_date
SELECT
  CASE
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 1 THEN '0-1 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 3 THEN '1-3 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 30 THEN '3-30 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 90 THEN '30-90 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 180 THEN '90-180 days'
    ELSE '180+ days'
  END as age_bucket,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb,
  ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 2) as percent_of_objects
FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
GROUP BY
  CASE
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 1 THEN '0-1 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 3 THEN '1-3 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 30 THEN '3-30 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 90 THEN '30-90 days'
    WHEN DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) < 180 THEN '90-180 days'
    ELSE '180+ days'
  END
ORDER BY
  CASE age_bucket
    WHEN '0-1 days' THEN 1
    WHEN '1-3 days' THEN 2
    WHEN '3-30 days' THEN 3
    WHEN '30-90 days' THEN 4
    WHEN '90-180 days' THEN 5
    ELSE 6
  END;
```

### Workflow 3: Identify Optimization Opportunities

**Step 1**: Query S3 Metadata for Standard storage > 30 days old (IA candidates)

```sql
-- Find objects in STANDARD storage class older than 30 days
-- Group by top-level prefix to identify patterns
SELECT
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key) - 1)
    ELSE key
  END as prefix,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb,
  ROUND(AVG(DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP)), 0) as avg_age_days,
  MIN(last_modified_date) as oldest_object,
  MAX(last_modified_date) as newest_object,
  ROUND(SUM(size) * 0.0125 / 1024.0 / 1024.0 / 1024.0, 2) as monthly_standard_cost,
  ROUND(SUM(size) * 0.0125 / 1024.0 / 1024.0 / 1024.0, 2) -
  ROUND(SUM(size) * 0.0125 / 1024.0 / 1024.0 / 1024.0, 2) as potential_savings_ia
FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
WHERE storage_class = 'STANDARD'
  AND DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) >= 30
GROUP BY
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key) - 1)
    ELSE key
  END
HAVING total_size_gb > 1  -- Only show meaningful opportunities (>1GB)
ORDER BY total_size_gb DESC
LIMIT 50;
```

**Step 3**: Query inventory for Standard storage > 90 days old (Glacier candidates)

```sql
-- Find objects in STANDARD storage class older than 90 days
-- These are candidates for Glacier Instant Retrieval or Glacier Flexible
SELECT
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key) - 1)
    ELSE key
  END as prefix,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb,
  ROUND(AVG(DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP)), 0) as avg_age_days,
  -- Cost estimates (approximate)
  ROUND(SUM(size) * 0.023 / 1024.0 / 1024.0 / 1024.0, 2) as monthly_standard_cost,
  ROUND(SUM(size) * 0.004 / 1024.0 / 1024.0 / 1024.0, 2) as monthly_glacier_ir_cost,
  ROUND(SUM(size) * 0.0036 / 1024.0 / 1024.0 / 1024.0, 2) as monthly_glacier_flex_cost,
  ROUND(
    (SUM(size) * 0.023 / 1024.0 / 1024.0 / 1024.0) -
    (SUM(size) * 0.004 / 1024.0 / 1024.0 / 1024.0),
    2
  ) as potential_monthly_savings_glacier_ir
FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
WHERE storage_class = 'STANDARD'
  AND DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) >= 90
GROUP BY
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key) - 1)
    ELSE key
  END
HAVING total_size_gb > 1
ORDER BY potential_monthly_savings_glacier_ir DESC
LIMIT 50;
```

**Step 4**: Evaluate Intelligent-Tiering candidates

```sql
-- Find objects suitable for Intelligent-Tiering
-- Criteria: STANDARD class, > 128KB, older than 30 days
SELECT
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key) - 1)
    ELSE key
  END as prefix,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as total_size_gb,
  ROUND(AVG(DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP)), 0) as avg_age_days,
  ROUND(100.0 * SUM(CASE WHEN size > 131072 THEN 1 ELSE 0 END) / COUNT(*), 1) as pct_over_128kb,
  -- Only meaningful if most objects are >128KB
  CASE
    WHEN 100.0 * SUM(CASE WHEN size > 131072 THEN 1 ELSE 0 END) / COUNT(*) > 80
    THEN 'Good IT candidate'
    ELSE 'Not suitable (too many small objects)'
  END as it_suitability
FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
WHERE storage_class = 'STANDARD'
  AND DATE_DIFF('day', last_modified_date, CURRENT_TIMESTAMP) >= 30
GROUP BY
  CASE
    WHEN POSITION('/' IN key) > 0
    THEN SUBSTRING(key, 1, POSITION('/' IN key) - 1)
    ELSE key
  END
HAVING total_size_gb > 1
ORDER BY total_size_gb DESC
LIMIT 50;
```

```sql
    size,
    CASE WHEN size > 131072 THEN 1 ELSE 0 END as is_over_128kb
  FROM "s3tables://arn:aws:s3tables:REGION:ACCOUNT_ID:bucket/BUCKET_S3METADATA/default/SOURCE_BUCKET-metadata"
)
SELECT
  prefix,
  storage_class,
  COUNT(*) as object_count,
  ROUND(SUM(size) / 1024.0 / 1024.0 / 1024.0, 2) as size_gb,
  ROUND(AVG(age_days), 0) as avg_age_days,
  CASE
    WHEN storage_class != 'STANDARD' THEN 'Already optimized'
    WHEN AVG(age_days) < 30 THEN 'Keep STANDARD (frequently accessed)'
    WHEN AVG(age_days) BETWEEN 30 AND 89 THEN 'Move to STANDARD_IA'
    WHEN AVG(age_days) BETWEEN 90 AND 179 THEN 'Move to GLACIER_IR'
    WHEN AVG(age_days) >= 180 AND 100.0 * SUM(is_over_128kb) / COUNT(*) > 80 THEN 'Move to INTELLIGENT_TIERING or GLACIER'
    WHEN AVG(age_days) >= 180 THEN 'Move to GLACIER_FLEXIBLE'
    ELSE 'Evaluate case-by-case'
  END as recommendation,
  ROUND(SUM(size) * 0.023 / 1024.0 / 1024.0 / 1024.0, 2) as current_monthly_cost_if_standard,
  CASE
    WHEN AVG(age_days) BETWEEN 30 AND 89
      THEN ROUND((SUM(size) * 0.023 - SUM(size) * 0.0125) / 1024.0 / 1024.0 / 1024.0, 2)
    WHEN AVG(age_days) BETWEEN 90 AND 179
      THEN ROUND((SUM(size) * 0.023 - SUM(size) * 0.004) / 1024.0 / 1024.0 / 1024.0, 2)
    WHEN AVG(age_days) >= 180
      THEN ROUND((SUM(size) * 0.023 - SUM(size) * 0.0036) / 1024.0 / 1024.0 / 1024.0, 2)
    ELSE 0
  END as potential_monthly_savings
FROM prefix_stats
GROUP BY prefix, storage_class
HAVING size_gb > 1
ORDER BY potential_monthly_savings DESC
LIMIT 100;
```

**Decision matrix**:

| Access Pattern | Storage Class | Criteria |
|----------------|---------------|----------|
| Frequent access | Standard | Accessed weekly+ |
| Infrequent access | Standard-IA | Not accessed 30+ days |
| Rare access | Glacier Instant | Not accessed 90+ days, <1min retrieval needed |
| Archive | Glacier Flexible | Not accessed 180+ days, hours retrieval OK |
| Deep archive | Glacier Deep | Not accessed 180+ days, 12hrs retrieval OK |
| Unpredictable | Intelligent-Tiering | Variable access, >128KB objects |

## Troubleshooting

### S3 Metadata not generating
```bash
# Calculate expected resources
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
ACCOUNT_ID_HASH=$(echo -n "${ACCOUNT_ID}" | sha256sum | cut -c1-7)
REGION=$(aws configure get region)
BUCKET_S3METADATA="ghlitch-${ACCOUNT_ID_HASH}-s3metadata"
SOURCE_BUCKET="your-bucket-name"
TABLE_NAME="${SOURCE_BUCKET}-metadata"

# Check S3 Tables bucket exists
aws s3tables list-table-buckets --region "${REGION}" | grep "${BUCKET_S3METADATA}" || echo "S3 Tables bucket not found"

# Check metadata configuration exists on source bucket
aws s3api get-bucket-metadata-table-configuration \
  --bucket "${SOURCE_BUCKET}" \
  --region "${REGION}"

# Check metadata table was created
aws s3tables get-table \
  --table-bucket-arn "arn:aws:s3tables:${REGION}:${ACCOUNT_ID}:bucket/${BUCKET_S3METADATA}" \
  --namespace "default" \
  --name "${TABLE_NAME}" \
  --region "${REGION}"
```

**If metadata table not found**:
- Wait 1 hour after enabling S3 Metadata for initial backfill
- Verify S3 Tables bucket exists and is in the same region as source bucket
- Check IAM permissions include `s3tables:CreateTable`

**If backfill not completing**:
- Check bucket size - very large buckets (billions of objects) may take longer
- Verify no IAM deny policies blocking S3 Tables service
- Check CloudTrail for S3 Metadata API errors

### Athena query returns no data
```bash
# Calculate expected resources
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
ACCOUNT_ID_HASH=$(echo -n "${ACCOUNT_ID}" | sha256sum | cut -c1-7)
REGION=$(aws configure get region)
SOURCE_BUCKET="your-bucket-name"
BUCKET_S3METADATA="ghlitch-${ACCOUNT_ID_HASH}-s3metadata"
TABLE_ARN="s3tables://arn:aws:s3tables:${REGION}:${ACCOUNT_ID}:bucket/${BUCKET_S3METADATA}/default/${SOURCE_BUCKET}-metadata"
ATHENA_OUTPUT="s3://ghlitch-${ACCOUNT_ID_HASH}-objects/budget/stats/athena-results/"

# Check metadata table exists
aws s3tables get-table \
  --table-bucket-arn "arn:aws:s3tables:${REGION}:${ACCOUNT_ID}:bucket/${BUCKET_S3METADATA}" \
  --namespace "default" \
  --name "${SOURCE_BUCKET}-metadata" \
  --region "${REGION}"

# Test simple query
aws athena start-query-execution \
  --query-string "SELECT COUNT(*) as total_objects FROM \"${TABLE_ARN}\"" \
  --result-configuration "OutputLocation=${ATHENA_OUTPUT}"
```

**If query returns 0 rows but objects exist**:
- Verify S3 Metadata backfill completed (wait 1 hour after enabling)
- Check that metadata table exists in S3 Tables (see previous troubleshooting step)
- Verify source bucket has objects: `aws s3 ls s3://${SOURCE_BUCKET} | head`

**If query fails with "Table not found"**:
- Verify TABLE_ARN is correctly formatted
- Check S3 Tables bucket name matches: `ghlitch-${ACCOUNT_ID_HASH}-s3metadata`
- Ensure metadata table name is `${SOURCE_BUCKET}-metadata`

### Athena query fails with S3 Tables errors
- Verify Athena engine version supports S3 Tables connector (Athena engine v3+)
- Check IAM permissions include `s3tables:GetTable`, `s3tables:GetTableData`
- Ensure S3 Tables bucket is in the same region as Athena workgroup

### Athena query is too expensive
- S3 Metadata uses Iceberg columnar format - already optimized
- Use `SELECT` specific columns instead of `SELECT *`
- Add `WHERE` clauses to filter data (Iceberg supports predicate pushdown)
- Use `LIMIT` when exploring data
- Iceberg's partition pruning automatically reduces scan size

**Cost optimization tips**:
- Iceberg only scans relevant data files (unlike full table scans)
- Metadata queries are very cheap (metadata is small)
- Column projection reduces I/O significantly
